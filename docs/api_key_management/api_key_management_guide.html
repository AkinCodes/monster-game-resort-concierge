01. Executive Summary

The API Key Management System transforms the APIKeyManager class from dead code into a fully wired, production-grade feature. It enables administrators to create, list, revoke, and audit managed API keys through dedicated REST endpoints, with full integration into the existing authentication flow.

What Changed

File Change Status
app/security.py Added list_keys, revoke_key, get_usage; removed dead code Modified
app/admin_routes.py Admin router with 4 CRUD endpoints New File
app/main.py Initialised APIKeyManager, mounted admin router Modified
app/auth_mixins.py Rewrote jwt_or_api_key with managed key support + security fixes Modified
tests/test_api_key_manager.py 16 tests (unit + integration) New File
tests/conftest.py Added new tables to cleanup list Modified

Key Capabilities Delivered

  • Key Creation with mr_ prefix, SHA-256 hashing, configurable expiry (1–365 days)
  • Key Verification during auth flow with automatic last_used_at tracking
  • Key Revocation via admin endpoint (soft-delete pattern)
  • Audit Logging of every managed-key authentication attempt (success and failure)
  • Security Hardening — removed hardcoded backdoor, fixed JWT verification bug

02. Why This Feature Exists

The Problem

Before this implementation, the Monster Resort Concierge had only one authentication mechanism for API clients: a single static API key stored in MRC_API_KEY. This created several problems:

Problem Impact
No key rotationIf the key leaked, every integration broke during rotation
No per-user attributionAll API consumers looked identical in logs
No expiryKeys lived forever once issued
No audit trailNo visibility into who called what, when
No revocationCompromised keys could not be individually disabled
Dead codeAPIKeyManager existed but was never instantiated

The Solution

Managed API keys give each integration partner or service their own key tied to a user_id — with automatic expiry, instant revocation, and a full audit trail. Create a new key, migrate the consumer, revoke the old key — zero downtime rotation.

Portfolio Value

This feature demonstrates production security patterns that interviewers and reviewers look for: defence-in-depth (multiple auth mechanisms), zero-trust key storage (only hash stored, raw key shown once), audit logging for compliance (SOC 2, GDPR), clean API design with Pydantic validation, and comprehensive test coverage.

Database Schema

Two tables are auto-created by APIKeyManager._ensure_schema() on first init:

-- Stores hashed keys and metadata
CREATE TABLE IF NOT EXISTS api_keys (
    key_hash      TEXT PRIMARY KEY,   -- SHA-256 of the raw key
    user_id       TEXT NOT NULL,      -- Owner identifier
    created_at    TEXT NOT NULL,      -- ISO 8601 timestamp
    expires_at    TEXT,               -- ISO 8601 (NULL = never)
    last_used_at  TEXT,               -- Updated on each verify
    is_active     INTEGER DEFAULT 1   -- 0 = revoked
);

-- Audit log of every authentication attempt
CREATE TABLE IF NOT EXISTS api_key_usage (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    key_hash  TEXT NOT NULL,        -- Links to api_keys.key_hash
    endpoint  TEXT NOT NULL,        -- e.g. "/chat"
    timestamp TEXT NOT NULL,        -- ISO 8601
    success   INTEGER NOT NULL      -- 1 = authenticated, 0 = rejected
);

Important: The raw API key is never stored. Only the SHA-256 hash is persisted. The raw key is returned exactly once during creation.

The core class that handles all key lifecycle operations.

Truncated key_id

External-facing methods use a truncated key_id (first 12 characters of the SHA-256 hash) instead of the full 64-character hash. 12 hex chars = 48 bits = 281 trillion possibilities — sufficient for identification and safe to display in dashboards.

4.2 Admin Routes app/admin_routes.py

A standalone APIRouter that provides CRUD operations for API keys. Mounted at /admin/api-keys.

  • Pydantic request/response models for all endpoints
  • get_api_key_manager(request) dependency extracts manager from app.state
  • All endpoints require auth via jwt_or_api_key
  • Appears as its own "admin" section in FastAPI auto-docs (/docs)

4.3 Auth Mixin app/auth_mixins.py

The jwt_or_api_key dependency was rewritten to support three authentication mechanisms:

Before

  • No managed key support
  • Hardcoded hex backdoor accepted as valid token
  • Called verify_token(token) which expected HTTPAuthorizationCredentials — JWT auth never actually worked
  • No audit logging
  • No Request parameter

After

  • Full mr_ managed key support via manager.verify_key()
  • Backdoor removed
  • Calls pyjwt.decode() directly — JWT auth works correctly
  • log_usage() on every managed-key attempt
  • Request parameter to access app.state.api_key_manager

4.4 App Wiring app/main.py

Four lines added to build_app():

# Import (at top of file)
from .security import install_rate_limiter, APIKeyManager
from .admin_routes import router as admin_router

# In build_app(), after db = DatabaseManager(settings):
api_key_manager = APIKeyManager(db)
app.state.api_key_manager = api_key_manager

# After install_rate_limiter(app, settings):
app.include_router(admin_router)

The APIKeyManager is stored on app.state (FastAPI's built-in mechanism for sharing objects across requests) rather than module-level globals — making it testable and explicit.

05. API Reference

All endpoints require authentication via jwt_or_api_key (static API key, managed key, or JWT).

POST /admin/api-keys

Create a new managed API key.

Request Body

FieldTypeRequiredDefaultConstraints
user_idstringYes—min 1 character
expires_daysintegerNo901–365

Example

curl -X POST http://localhost:8000/admin/api-keys \
  -H "Authorization: Bearer <your-api-key>" \
  -H "Content-Type: application/json" \
  -d '{"user_id": "mobile-app-v2", "expires_days": 30}'

Response (200)

{
  "raw_key": "mr_K7xP2mN9qR4wT6yB1cD8eF3gH5jL0nM2pQ4sU6vX8zA",
  "user_id": "mobile-app-v2",
  "expires_days": 30,
  "message": "Store this key securely - it will not be shown again."
}

Important: The raw_key is only returned once. It cannot be retrieved later.

GET /admin/api-keys

List all managed keys (metadata only). Filterable by ?user_id=.

Example

# List all keys
curl http://localhost:8000/admin/api-keys \
  -H "Authorization: Bearer <your-api-key>"

# Filter by user
curl "http://localhost:8000/admin/api-keys?user_id=mobile-app-v2" \
  -H "Authorization: Bearer <your-api-key>"

Response (200)

[
  {
    "key_id": "a1b2c3d4e5f6",
    "user_id": "mobile-app-v2",
    "created_at": "2026-02-13T10:30:00.000000",
    "expires_at": "2026-03-15T10:30:00.000000",
    "last_used_at": "2026-02-13T14:22:00.000000",
    "is_active": true
  }
]

DELETE /admin/api-keys/{key_id}

Revoke a key (soft delete). Use the key_id from the list endpoint.

Example

curl -X DELETE http://localhost:8000/admin/api-keys/a1b2c3d4e5f6 \
  -H "Authorization: Bearer <your-api-key>"

Response (200)

{ "ok": true, "detail": "Key a1b2c3d4e5f6 revoked" }

Response (404)

{ "detail": "Key not found" }

GET /admin/api-keys/usage

View the audit log. Filterable by ?key_id= and ?limit=.

Example

curl "http://localhost:8000/admin/api-keys/usage?limit=10" \
  -H "Authorization: Bearer <your-api-key>"

Response (200)

[
  {
    "id": 42,
    "key_id": "a1b2c3d4e5f6",
    "user_id": "mobile-app-v2",
    "endpoint": "/chat",
    "timestamp": "2026-02-13T14:22:00.000000",
    "success": true
  }
]

06. Security Design Decisions

6.1 Key Storage: Hash-Only

Raw keys are never stored. The system uses a one-way SHA-256 hash.

1

Creation

Generate mr_K7xP2mN9qR4w... → SHA-256 → Store hash in DB → Return raw key to caller once

2

Verification

Receive mr_K7xP2mN9qR4w... → SHA-256 → DB lookup by hash → Check active + not expired → Return user_id

Why? If the database is compromised, attackers get hashes, not usable keys. SHA-256 is one-way; they cannot reverse it.

6.2 Key Prefix: mr_

  • Quick identification in logs and config files
  • Routing — auth flow only calls verify_key() for mr_ tokens, avoiding unnecessary DB lookups
  • Grep-friendly — grep "mr_" .env instantly finds managed keys

6.3 Soft Revocation

revoke_key() sets is_active = 0 rather than deleting. This preserves audit history, forensic evidence (creation date, owner, revocation time), and allows admin recovery.

6.4 Audit on Failure

Both successful and failed authentication attempts are logged. Failed attempts are often more important than successes for security monitoring — repeated failures suggest a leaked/compromised key being brute-forced.

10. Troubleshooting

"API key manager not available" (503)

Cause: app.state.api_key_manager is None.

Fix: Ensure main.py has: api_key_manager = APIKeyManager(db) and app.state.api_key_manager = api_key_manager

Managed key returns 401 even though it was just created

Check in order:

  1. Prefix — Key must start with mr_
  2. Expiry — Check expires_at in key listing
  3. Revoked — Check is_active is true
  4. Typo — Copy-paste the full key exactly as returned

11. Security Fixes Included

11.1 Hardcoded Backdoor Removed

Before (auth_mixins.py:27)

if (
  token == settings.api_key.strip()
  or token == "6f2b8e3a9d..."
):
    return "api_key_user"

Anyone who discovered this hex string could bypass authentication.

After

if token == settings.api_key.strip():
    return "api_key_user"

Only the configured MRC_API_KEY is accepted.

11.2 JWT Verification Bug Fixed

Before

# token is a plain string
user = verify_token(token)
# verify_token expects HTTPAuth
# CredentialsCredentials
# -> AttributeError -> silently
# caught -> JWT never worked

After

payload = pyjwt.decode(
    token, SECRET_KEY,
    algorithms=[ALGORITHM]
)
username = payload.get("sub")
# JWT tokens are correctly
# decoded and validated
