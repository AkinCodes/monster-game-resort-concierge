The Engine Room taught you how the hotel's infrastructure works — databases, configuration, the plumbing that keeps everything running.

Now: The hotel needs to communicate with the outside world. There are three phone lines, each run by a different operator who speaks a slightly different dialect. But every answer that comes back into the hotel arrives on one standard form. If a line goes down, the Head Operator silently tries the next one. The guest never hears static.

The Standard Forms

Before any operator picks up the phone, the hotel needs a universal language. These four things — three paper forms and one set of rules — make sure every answer looks the same, no matter who answered the call.

Big Bird — The Form Designer

Big Bird sits in the back office with a stack of blank paper. He designs the standard message form that every operator in the building must use.

"It doesn't matter which phone line you use — when a message comes back, it must be on THIS form."

The form has four boxes: who's speaking (the role), what they said (the content), any service requests (tool calls — like asking to book a room mid-call), and how long the call lasted (usage — how many words went in, how many came back).

Big Bird also writes a single rule on the wall: "Every operator must know how to chat and how to translate service request formats. If you can't do those two things, you can't sit at a phone."

💡 The standard forms are the reason the rest of the hotel never needs to know which operator answered. The War Room, the Night Watchman, the Guest Book — they all read the same form. Whether Line 1, Line 2, or Line 3 picked up, the form that comes back looks identical. That is the entire point of the abstraction.
"Three operators, three dialects, one form. The hotel reads the form — never the dialect."

Operator #1 — The Standard Dialect

Operator #1 speaks the standard dialect. Messages go straight through with no translation needed. When service requests come back, they are already in the right format. This is the baseline everyone else is measured against.

Operator #1 — The Native Speaker

Operator #1 sits at Line 1. When you hand him a stack of messages, he passes them straight to the outside world. No translation needed — the dialect the hotel uses internally is the same one this operator speaks natively.

When the answer comes back, Operator #1 reads through the response and fills in the standard form: the content goes in the "what they said" box, any service requests go in the "tool calls" box, and the call length goes in the "usage" box.

The service request format is also native. When the hotel says "here are the service desks the guest can use," Operator #1 just takes the list as-is. Zero translation in either direction.

OpenAIProvider class OpenAIProvider(LLMProvider): def __init__(self, api_key: str, model: str = "gpt-4o-mini"): import openai self._client = openai.AsyncOpenAI(api_key=api_key) self._model = model def translate_tool_schemas(self, openai_schemas): return openai_schemas # native format — no translation! def _to_openai_messages(self, messages): out = [] for m in messages: msg = {"role": m.role, "content": m.content} if m.tool_calls: # attach service requests msg["tool_calls"] = [ {"id": tc.id, "type": "function", "function": {"name": tc.name, "arguments": tc.arguments}} for tc in m.tool_calls ] if m.tool_call_id: # link answer to its request msg["tool_call_id"] = m.tool_call_id out.append(msg) return out
OpenAIProvider.chat() async def chat(self, messages, tools=None, model=None): kwargs = { "model": model or self._model, "messages": self._to_openai_messages(messages), } if tools: # hand over the service desk list kwargs["tools"] = self.translate_tool_schemas(tools) kwargs["tool_choice"] = "auto" resp = await self._client.chat.completions.create(**kwargs) msg = resp.choices[0].message # --- Parse service requests from the response --- tool_calls = [] if getattr(msg, "tool_calls", None): tool_calls = [ LLMToolCall(id=tc.id, name=tc.function.name, arguments=tc.function.arguments) for tc in msg.tool_calls ] # --- Parse call duration (usage) --- usage = {} if resp.usage: usage = { "prompt_tokens": resp.usage.prompt_tokens, "completion_tokens": resp.usage.completion_tokens, "total_tokens": resp.usage.total_tokens, } # --- Fill in the standard form and return it --- return LLMResponse( content=msg.content or "", tool_calls=tool_calls, model=resp.model, provider="openai", usage=usage, )
💡 Operator #1 is the simplest because the hotel's internal format was modeled after his dialect. The tool schemas pass through untouched. The messages need only light reshaping (adding the "function" wrapper on tool calls). The response parsing is straightforward: pull content, pull tool_calls, pull usage, fill the form.
"Line 1 speaks the house language. No phrasebook needed."

Operator #2 — The Translator

Operator #2 speaks a different dialect. The hotel must translate messages in both directions. Two key differences make this operator the most educational one in the building.

Operator #2 — The Foreign Correspondent

Operator #2 is brilliant but particular. Two things about his dialect are fundamentally different from the house language:

Difference #1: "The house rules must be sent separately." In Operator #1's dialect, the house rules (system message) are mixed right into the conversation, just another line on the notepad. Operator #2 insists: "Give me the house rules in a separate envelope." So the translator must scan every message, pull out any house rules, and hand them over separately.

Difference #2: "Service requests look different." Operator #1 calls them "function calls." Operator #2 calls them "tool use blocks." The form layout is different — the name goes in a different box, the details are structured differently. The translator must reformat them in both directions: outgoing requests AND incoming responses.

translate_tool_schemas() def translate_tool_schemas(self, openai_schemas): # Translate the hotel's service desk list into Operator #2's dialect anthropic_tools = [] for schema in openai_schemas: fn = schema.get("function", {}) anthropic_tools.append({ "name": fn.get("name", ""), # same name, different envelope "description": fn.get("description", ""), "input_schema": fn.get("parameters", { # "parameters" becomes "input_schema" "type": "object", "properties": {} }), }) return anthropic_tools
_to_anthropic_messages() — The System Extraction def _to_anthropic_messages(self, messages): # Scan every message. Pull out house rules. Reformat the rest. system_text = "" anthropic_msgs = [] for m in messages: if m.role == "system": # House rules? Separate envelope! system_text += m.content + "\n" continue if m.role == "tool": # Tool result? Reformat as user msg anthropic_msgs.append({ "role": "user", # Op #2 sees tool results as "user" "content": [{ "type": "tool_result", "tool_use_id": m.tool_call_id or "", "content": m.content, }], }) continue if m.role == "assistant" and m.tool_calls: # Assistant with service requests content_blocks = [] if m.content: content_blocks.append({"type": "text", "text": m.content}) for tc in m.tool_calls: content_blocks.append({ "type": "tool_use", # "function" becomes "tool_use" "id": tc.id, "name": tc.name, "input": json.loads(tc.arguments), # string -> dict }) anthropic_msgs.append({"role": "assistant", "content": content_blocks}) continue anthropic_msgs.append({"role": m.role, "content": m.content}) return system_text.strip(), anthropic_msgs
AnthropicProvider.chat() async def chat(self, messages, tools=None, model=None): # Step 1: Translate outgoing messages (extract system, reformat) system_text, anthropic_msgs = self._to_anthropic_messages(messages) kwargs = { "model": model or self._model, "max_tokens": 4096, "messages": anthropic_msgs, } if system_text: # hand over the separate envelope kwargs["system"] = system_text if tools: kwargs["tools"] = self.translate_tool_schemas(tools) resp = await self._client.messages.create(**kwargs) # Step 2: Translate incoming response blocks to standard form content_text = "" tool_calls = [] for block in resp.content: if block.type == "text": content_text += block.text elif block.type == "tool_use": # "tool_use" back to LLMToolCall tool_calls.append(LLMToolCall( id=block.id, name=block.name, arguments=json.dumps(block.input), # dict -> string )) # Step 3: Translate usage (different field names!) usage = {} if resp.usage: usage = { "prompt_tokens": resp.usage.input_tokens, # different name! "completion_tokens": resp.usage.output_tokens, # different name! "total_tokens": resp.usage.input_tokens + resp.usage.output_tokens, } return LLMResponse( content=content_text, tool_calls=tool_calls, model=resp.model, provider="anthropic", usage=usage, )
💡 Operator #2 is the most educational. If you understand the translation — the system message extraction, the tool_use vs function mapping, the input_tokens vs prompt_tokens rename — you understand why the standard forms exist. The entire point is that the War Room hands over messages in one format, and Operator #2 quietly reshapes them before sending, then reshapes the response back. The War Room never knows any of this happened.
"Same conversation, different envelope. The translator handles it all."

The Head Operator

Operator #3 sits in the basement — the local switchboard. And above all three operators sits the Head Operator, whose only job is: try Line 1, then Line 2, then Line 3. The guest hears an answer — never static.

Operator #3 — The Basement Switchboard

Operator #3 is different from the others. Lines 1 and 2 are long-distance calls — they go out over the wire to faraway exchanges. Operator #3 uses a direct wire to the basement — a local switchboard running right inside the hotel building.

He speaks the same dialect as Operator #1 (no translation needed), but he uses a raw cable (HTTP) instead of the fancy long-distance telephone service. No call charges — it is free. But the connection is slower, and the voice on the other end is less polished.

The basement switchboard has a generous timeout — 120 seconds — because local calls sometimes take a while to connect.

OllamaProvider class OllamaProvider(LLMProvider): def __init__(self, base_url="http://localhost:11434", model="llama3"): self._base_url = base_url.rstrip("/") self._model = model self._http = httpx.AsyncClient(timeout=120.0) # 2 min — local is slow def translate_tool_schemas(self, openai_schemas): return openai_schemas # same dialect as Operator #1 async def chat(self, messages, tools=None, model=None): payload = { "model": model or self._model, "messages": self._to_ollama_messages(messages), "stream": False, # wait for full answer, no partial } if tools: payload["tools"] = self.translate_tool_schemas(tools) # Raw HTTP call to the basement switchboard resp = await self._http.post( f"{self._base_url}/api/chat", json=payload ) resp.raise_for_status() data = resp.json() msg = data.get("message", {}) # Parse tool calls from the response (if any) tool_calls = [] for tc in msg.get("tool_calls", []): fn = tc.get("function", {}) tool_calls.append(LLMToolCall( id=tc.get("id", ""), name=fn.get("name", ""), arguments=json.dumps(fn.get("arguments", {})) if isinstance(fn.get("arguments"), dict) else fn.get("arguments", "{}"), )) return LLMResponse( content=msg.get("content", ""), tool_calls=tool_calls, model=payload["model"], provider="ollama", usage={}, # local switchboard doesn't track call length )
The Head Operator — ModelRouter

The Head Operator sits above all three lines. He does not speak any dialect himself — he has no phone. His only job is management.

When a call needs to go out, he tries Line 1 first. If it is busy or down, he tries Line 2. Then Line 3. The guest hears an answer — never static.

If ALL three lines are down — an event so rare it has never happened — only then does the Head Operator walk to the guest and say: "I'm sorry. All lines are down."

He can also be told: "Don't try the next line." If fallback is disabled, when Line 1 goes down, the Head Operator immediately reports the failure instead of trying Line 2.

ModelRouter — The Head Operator class ModelRouter: def __init__(self, providers: List[LLMProvider], fallback_enabled=True): self.providers = providers # The three phone lines, in priority order self.fallback_enabled = fallback_enabled async def chat(self, messages, tools=None, model=None): last_error = None for provider in self.providers: # Try each line in order try: logger.info(f"Trying LLM provider: {provider.name}") response = await provider.chat(messages, tools, model) logger.info(f"LLM provider {provider.name} succeeded") return response # Got an answer! Return it. except Exception as e: last_error = e logger.warning(f"LLM provider {provider.name} failed: {e}") if not self.fallback_enabled: # No fallback? Stop here. raise # All lines down. Report failure. raise RuntimeError(f"All LLM providers failed. Last error: {last_error}")

⚠️ What Happens When Line 1 Goes Down?

📨 A call needs to go out. The Head Operator picks up Line 1 (OpenAI).
⚡ Line 1 is down! An exception is raised. The Head Operator catches it.
📝 He writes a warning in the log: "LLM provider openai failed: Connection refused"
🔄 Fallback is enabled, so the loop continues. He picks up Line 2 (Anthropic).
✅ Line 2 answers! Operator #2 translates the message, makes the call, translates the response, fills in the standard form.
📋 The standard form goes back to the hotel. The guest never knew Line 1 was down.

💡 The Head Operator's power comes from simplicity. He does not translate. He does not speak any dialect. He simply has a list of operators and a for-loop. Try each one. If it works, return the answer. If it fails and fallback is on, try the next. If all fail, admit defeat. That is the entire ModelRouter — a for-loop with a try/except.
