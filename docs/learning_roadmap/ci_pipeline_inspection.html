The Assembly Line — What a CI Pipeline Actually Does

Every morning, a delivery truck pulls up. Inside is last night's work — code changes from the cleaning crew. Before any guest sees the hotel, the inspection team runs through a strict sequence of checks. Each inspector hands off to the next. If anyone raises a red flag, the line stops. The hotel stays closed.

STEP 1
Checkout
Big Bird receives the delivery truck
▾
Big Bird — The Receiving Dock

The overnight cleaning crew finished their shift. Now Big Bird needs to see what they actually did. The delivery truck backs up to the loading dock — that's the git push. Big Bird opens the back, pulls out every box, and lays them out on the dock so the inspectors can look at everything.

Without this step, there's nothing to inspect. Big Bird doesn't judge the contents. He just makes sure everything that was shipped is now on the table.

YAML # Pull the code onto the inspection table - uses: actions/checkout@v4
STEP 2
Install
The kitchen stocks up on ingredients
▾
The Kitchen — Stocking the Pantry

Before anyone can cook, the kitchen needs ingredients. pip install is the supply truck unloading flour, butter, eggs — every dependency your code needs to run. If the pantry is empty, the chef can't make anything. If a package is missing or the wrong version, the recipe fails.

This step reads the recipe list (requirements.txt) and fills the pantry with exactly what's needed. Nothing more, nothing less.

YAML - name: Install dependencies run: pip install -r requirements.txt
STEP 3
Lint
Cookie Monster inspects every room for crumbs
▾
Cookie Monster — The Crumb Inspector

"Me walk EVERY room. Me check EVERY corner. One crumb? FAILED."

Cookie Monster doesn't care if the room looks nice overall. He's not checking if the bed is comfortable or the view is good. He's looking for crumbs — unused imports, missing whitespace, lines that are too long, variables that were never used. Tiny messes that show sloppy work.

He walks into app/, checks every file. Then tests/, checks every file there too. If he finds even one crumb, he slams the clipboard down. FAILED. The line stops here.

YAML - name: Lint run: flake8 app tests
STEP 4
Test
Grover tests every faucet and lock
▾
Grover — The Meticulous Tester

"I, Grover, shall test each room VERY carefully."

Grover goes room by room. He turns on every faucet — does water come out? He tries every lock — does it click shut? He flips every light switch — does the bulb turn on? He doesn't just glance at the room. He uses everything the way a guest would.

If a faucet sprays sideways, that's a failure. If a lock doesn't catch, that's a failure. pytest is Grover walking through every room in the hotel, trying every single thing a guest might try, and writing down what broke.

The --cov flag is Grover's checklist: it tells you which rooms he visited and which ones he skipped. If coverage is low, there are rooms nobody tested.

YAML - name: Run tests run: pytest --cov=app --cov-report=term-missing env: MRC_DATABASE_URL: sqlite:///./test_ci.db MRC_OPENAI_API_KEY: dummy
STEP 5
Result
Count von Count tallies. Oscar writes the report.
▾
Count von Count + Oscar — The Verdict

Count von Count stands at the end of the line with his clipboard. He tallies every test result Grover handed him.

"ONE hundred four tests passed! ZERO failed! Ah ah ah!"

If everything passed, the green checkmark goes up. The hotel opens. Guests can enter.

But if even one test failed? Oscar crawls out of his trash can with the rejection stamp:

"REJECTED. Fix your mess."

The hotel stays closed. The delivery truck gets sent back. Nobody gets in until the crew fixes what's broken and sends a new shipment.

💡 Sequential gates, not parallel checks. Steps run in order. If step 3 (lint) fails, step 4 (test) never runs. The pipeline is sequential gates, not parallel checks. Each gate must open before the next inspector even gets to look. This is why a single crumb from Cookie Monster can block the entire hotel from opening — Grover never even gets to start his faucet tests.

The Morning Inspection Catches a Stowaway

Cookie Monster finds unclaimed luggage. The hotel stays closed.

Cookie Monster — Room-by-Room Sweep

Cookie Monster arrives for the morning inspection. Clipboard in hand, he walks room by room through app/database/db.py. He opens the closet and spots a suitcase: import re. He checks the guest registry — nobody in this room uses re. No re.match(), no re.sub(), no re.findall(). Nobody claimed this bag.

"This is STOWAWAY LUGGAGE! Somebody left this behind and never came back for it. INSPECTION FAILED!" Cookie Monster slams the clipboard shut. The hotel stays closed. No guest sees the lobby today.

Big Bird — But I Ran the Tests!

Big Bird protests: "But I ran all the tests locally! 104 tests passed! Every single one green!" Oscar pops out of his trash can next to the CI server: "You ran TESTS. You didn't run INSPECTION. Inspection comes FIRST. Cookie Monster walks the halls before Grover ever turns on a faucet. You skipped the walk."

app/database/db.py from __future__ import annotations import re # THE STOWAWAY — used nowhere import uuid from sqlalchemy import create_engine, text
CI OUTPUT $ flake8 app tests app/database/db.py:3:1: F401 're' imported but unused ##[error]Process completed with exit code 1. Step "Lint" FAILED Step "Run tests" SKIPPED — previous step failed
💡 Lint runs BEFORE tests. A lint failure (exit code 1) kills the pipeline. Tests never even wake up. Always run flake8 app tests && pytest locally — the && mirrors CI behavior. The double-ampersand means "only run the next command if the previous one succeeded." If lint fails, pytest never starts — exactly like CI.
The Fix — Remove the Stowaway

The fix is surgical. No test changes, no logic changes. Just remove the line that doesn't belong.

app/database/db.py — FIXED from __future__ import annotations # import re ← REMOVED — the stowaway is gone import uuid from sqlalchemy import create_engine, text
🎙️ Interview takeaway: CI pipelines run steps sequentially. A linter failure aborts the job before tests execute. The fix is the lint error, not the tests. "But the tests pass!" is not a defense — if lint fails, tests are never reached. Local dev should mirror CI order: flake8 && pytest. Never run just pytest and assume CI will pass.

The Guest Who Only Exists at Night

Grover runs the census. But some guests only check in when the moon is up.

Grover — The Night Shift Census

Grover works the night shift in CI. He walks room by room, checking that every registered guest is actually present. He reaches Room 14 — test_ragas_eval. The guest registry says MRC_OPENAI_API_KEY = "dummy". The RAGAS evaluation needs a real key to call OpenAI. With a dummy key, the call fails silently or the test gets skipped entirely.

"This guest is not checking in tonight. SKIP!" Grover marks Room 14 as empty and moves on. The pipeline glows green. Everything looks fine.

Big Bird — The Daytime Surprise

Daytime. Big Bird runs pytest locally. His .env has a real API key. Grover opens Room 14 — the guest IS here. The test runs, calls real OpenAI, but the assertion fails because the RAGAS library updated and context_precision is no longer in the results. The metric got renamed upstream.

Same code. Same tests. Different results. CI says green. Local says red. Who do you trust?

🌙 CI (Night)

MRC_OPENAI_API_KEY: dummy

→ Test skipped or fails silently

→ Pipeline GREEN

☀️ Local (Day)

MRC_OPENAI_API_KEY: sk-real-...

→ Test runs → assertion error

→ RED locally

ci.yml — ENV VARS env: MRC_OPENAI_API_KEY: dummy MRC_API_KEY: dummy MRC_DATABASE_URL: sqlite:///./test_ci.db
test_ragas.py def test_ragas_eval_example(): results = evaluate_rag_batch(samples) assert "faithfulness" in results # passes assert "context_precision" in results # FAILS — metric renamed upstream
💡 A green CI badge doesn't mean your code works. It means it passed the tests that actually ran. Skipped tests are invisible danger. Audit pytest -v output for SKIPPED counts. If you see "14 passed, 3 skipped" — those 3 skipped tests are rooms Grover never entered. You have no idea what's inside.
🎙️ Interview takeaway: Tests that depend on environment variables create divergence between CI and local. The same test suite produces different results depending on which secrets are present. Test BOTH code paths, or use markers to separate integration from unit tests — @pytest.mark.integration lets you run pytest -m "not integration" in CI and pytest -m integration locally with real keys.

Two Inspectors Doing the Same Job

Two workflow files. Same corridor. Double the invoices.

Oscar — The Budget Auditor

Two night watchmen — Inspector CI and Inspector Test Suite — patrol the same corridor. Both check the locks. Both count the towels. Both file reports. The hotel gets two bills for the same work.

Oscar crawls out of his trash can, waving two invoices: "WHY are we paying TWO people to check the SAME locks?!"

He's right. Two workflow files. Same trigger. Same steps. Double the compute time. Double the cost. And if one passes and the other fails because of a trivial difference? Now you're debugging a phantom problem that only exists because you have two inspectors with slightly different clipboards.

What they do ci.yml test.yml
Trigger push to main push to main
Python setup actions/setup-python@v5 actions/setup-python@v5
Install deps pip install -r requirements.txt pip install -r requirements.txt
Pip caching Yes (actions/cache) No
Lint flake8 flake8
Test pytest --cov=app pytest tests/ -v --cov=app
DIFF # Lint — identical behavior, cosmetic difference ci.yml: flake8 app tests test.yml: flake8 app/ tests/ ← trailing slashes, identical behavior # Test — same tests, one adds -v ci.yml: pytest --cov=app test.yml: pytest tests/ -v --cov=app ← adds -v, same tests
💸 Every workflow file is a separate inspector on the payroll. If two inspectors check the same things, you're paying double for nothing. GitHub Actions bills by the minute. Two identical workflows on the same trigger means every push costs twice as much — and you get zero extra safety from it.

🔧 The Fix — One Inspector, One Corridor

Delete the duplicate. Consolidate into a single workflow that does everything:

YAML — consolidated ci.yml name: CI on: push: branches: [main] pull_request: branches: [main] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: "3.12" cache: "pip" - run: pip install -r requirements.txt - run: flake8 app tests - run: pytest --cov=app --cov-report=term-missing -v
🎙️ Interview takeaway: Multiple workflows make sense when they serve different purposes — test vs deploy vs security scan. Two workflows doing the same thing on the same trigger waste compute. If you see duplicate workflows, the answer is always: consolidate, don't duplicate.

The Luggage Mismatch

Two guest lists that disagree. The day manager packed different bags than the night manager.

Big Bird — The Confused Front Desk

Two front-desk managers are arguing. Day Manager (pyproject.toml): "I have openai in Room 72 and anthropic in Room 73."

Night Manager (requirements.txt): "I don't have those. But I DO have psycopg2-binary in Room 79 and redis in Room 84."

Big Bird stares at both lists, feathers ruffled: "This is madness. We have TWO guest lists and they don't agree."

Which list is correct? Neither? Both? The hotel opens with whatever the night manager has — because CI uses requirements.txt. Meanwhile the day manager's list has packages that never get tested. And the night manager has packages that aren't in the official manifest. Nobody knows what's actually running in production.

Category Package Status
pyproject.toml ONLY openai Not tested in CI
pyproject.toml ONLY anthropic Not tested in CI
pyproject.toml ONLY mlflow Not tested in CI
pyproject.toml ONLY jupyter Not tested in CI
requirements.txt ONLY psycopg2-binary Not in project manifest
requirements.txt ONLY redis Not in project manifest
Version drift pandas >=2.3.3 vs >=2.0.0
⚠️ Two sources of truth is zero sources of truth. When pyproject.toml says one thing and requirements.txt says another, you have no idea what's actually installed. The dev environment, CI, and production could all be running different packages. One must be generated from the other — never hand-edit both.

🔧 The Fix — One Guest List, One Source

Pick one canonical source and derive the other:

  1. Option A — Generate requirements.txt from pyproject.toml:
    uv pip compile pyproject.toml -o requirements.txt
    This locks every version and ensures CI installs exactly what the project declares.
  2. Option B — Use pyproject.toml directly in CI:
    pip install ".[dev]"
    Skip requirements.txt entirely. Let the project manifest be the single source.
YAML — Option A # In your CI workflow - name: Install dependencies run: | pip install uv uv pip compile pyproject.toml -o requirements.txt pip install -r requirements.txt
YAML — Option B # Simpler — use pyproject.toml as the source - name: Install dependencies run: pip install ".[dev]"
🎙️ Interview takeaway: Modern Python uses pyproject.toml as the canonical source. If CI needs requirements.txt, generate it — never hand-edit both. When someone asks "how do you manage dependencies?", the answer is: one source of truth, lock files for reproducibility, and CI installs from the lock file.
