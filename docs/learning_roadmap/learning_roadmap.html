The Skeleton
Read these first · 4 files · ~1 hr
▼
These four files are the backbone. Every other file in the project exists to serve one of them. Read config first (it sets up all providers, thresholds, and feature flags), then database, then memory (how conversations persist), then main. After this phase you'll understand: what the app does, how data is stored, and how a request flows end to end.
⚙️
app/config.py Read First
All settings live here — API keys for OpenAI/Anthropic, Ollama config, LLM provider priority and fallback, hallucination thresholds, MLflow tracking, database path, and more (~160 lines). The central nervous system for every feature flag and credential.
Why: Everything references config. Reading it first means nothing else will confuse you.
🗄️
app/database.py Read Second
SQLite database. Creates the bookings table, handles create/read/update/delete. This is where guest bookings actually get saved to disk.
Why: main.py calls into this when someone books a room. You need to know what "saving a booking" actually means.
💭
app/memory.py Read Third
MemoryStore class. Manages conversation sessions and message history in the database. Auto-summarises old messages when a session gets too long, keeping context window usage efficient.
Why: Every /chat call reads and writes through MemoryStore. Without this, the AI has no memory of what you said 30 seconds ago. main.py depends on it directly.
🏗️
app/main.py The Big One
The FastAPI server. Routes, the agent loop (where the LLM decides what to do via ModelRouter), the system prompt, hallucination scoring, MLflow logging. This is the orchestrator — it pulls in every other module.
Why: Focus on the /chat route and the agent loop. Notice how _build_router() sets up multi-provider LLM routing, and how every response gets a confidence score from HallucinationDetector.
2
The Brain
How the AI actually works · 8 files · ~3 hrs
▼
This is the RAG + LLM orchestration + tool calling layer — the part that makes it an AI system instead of just a website. Start with llm_providers.py (how the app talks to any LLM), then read rag.py before advanced_rag.py (parent → child). Then langchain_rag.py (the LangChain alternative for benchmarking). Finally tools.py to see what actions the AI can take.
🔀
app/llm_providers.py Read First
Multi-model LLM abstraction. Defines LLMMessage/LLMResponse (normalised formats), LLMProvider ABC, and three implementations: OpenAIProvider, AnthropicProvider, OllamaProvider. ModelRouter tries providers in priority order with automatic fallback.
Why: Every LLM call in the app flows through this file. main.py's agent loop depends entirely on ModelRouter. Without reading this, you won't understand how chat requests reach any LLM.
🧬
app/rag.py Foundation
VectorRAG class. Sets up ChromaDB, embeds documents with HuggingFace, does semantic search. This is the base class that advanced_rag inherits from.
Why: advanced_rag.py calls super() constantly. If you skip this, advanced_rag makes no sense.
🧠
app/advanced_rag.py The Upgrade
Adds BM25 keyword search, Reciprocal Rank Fusion, and BGE cross-encoder reranking on top of VectorRAG. Three-stage pipeline: retrieve → fuse → rerank.
Why: This is what made the project go from "generic chatbot" to "actually accurate." The hybrid search is your biggest technical achievement.
🔗
app/langchain_rag.py Benchmark
LangChain-based RAG with the same interface as AdvancedRAG (ingest_texts, ingest_folder, search). Uses LangChain's Chroma + HuggingFaceEmbeddings + RecursiveCharacterTextSplitter. Exists for head-to-head benchmarking.
Why: Shows why you'd build custom RAG instead of using LangChain. Read this after advanced_rag.py to compare the two approaches.
🔧
app/tools.py Tool System
ToolRegistry class + all the tools: book_room, get_booking, generate_pdf, search_knowledge. Each tool has a JSON schema that tells the LLM how to call it.
Why: When the AI decides to "book a room," this file is where that decision becomes a real database operation. It's the bridge between language and action.
📄
data/knowledge/ The Knowledge
4 .txt files (63KB total): amenities, policies, FAQ, seasonal events. This is what the RAG system searches through to answer questions.
Why: Read one file fully. It shows you exactly what information the AI has access to — and why it can't answer things that aren't in here.
📥
app/ingest_knowledge.py CLI Tool
Command-line script for ingesting knowledge documents into the RAG vector store. Takes a folder path, initialises VectorRAG, and chunks + embeds all .txt files into ChromaDB.
Why: This is how data gets INTO the RAG system. main.py calls rag.ingest_folder() at startup, but this script lets you re-ingest manually.
🔄
app/refresh_store.py
Minimal in-memory dictionary for storing refresh tokens. A development/demo utility — would be replaced by Redis or a database table in production.

The Guardrails
Security, auth, caching, observability, hallucination detection · 13 files · ~3.5 hrs
▼
Production systems need more than "it works." These files handle security, authentication, performance, visibility, and response quality. Each does one job well. They're plugged into main.py as middleware, dependencies, or scoring layers. Read the auth chain first (auth.py → auth_mixins.py → security.py → admin_routes.py) to understand the full authentication picture, then the observability stack.
🔑
app/auth.py Auth Chain 1/4
JWT authentication. Creates and verifies JSON Web Tokens with HS256 + bcrypt password hashing. Enforces strong secret key at startup (min 32 chars, rejects default placeholders). Defines create_access_token, verify_token, hash_password, verify_password.
Why: This is authentication for human users (login flow). Read this before auth_mixins.py — it provides the JWT primitives that the mixin calls.
🔀
app/auth_mixins.py Auth Chain 2/4
The jwt_or_api_key dependency — FastAPI's auth gatekeeper. Checks X-API-Key header (static key, then managed mr_ keys) and Bearer tokens (static, managed, then JWT). Logs managed-key usage for audit trail. Every protected route depends on this.
Why: This is the single function that decides "are you allowed in?" Understanding the priority chain (static → managed → JWT) is key to the whole auth architecture.
🔐
app/security.py Auth Chain 3/4
Two responsibilities: (1) APIKeyManager class — creates, verifies, revokes, and audits managed API keys with SHA-256 hashing, expiry, and soft-delete revocation. (2) Rate limiting via slowapi (60 req/min per IP). The manager stores only hashes, never raw keys.
Why: The APIKeyManager is a complete key lifecycle system — creation, rotation, revocation, audit logging. Portfolio-worthy security pattern. Rate limiting is the second defense layer.
🛠️
app/admin_routes.py Auth Chain 4/4
Admin REST API for API key management. POST /admin/api-keys (create), GET (list), DELETE (revoke), GET /usage (audit log). Pydantic request/response models. All endpoints require auth. Extracts APIKeyManager from app.state.
Why: The admin-facing surface for the APIKeyManager. Shows how to build a separate admin router with its own prefix, dependency injection, and Pydantic validation.
👤
app/users_db.py
In-memory user database for demo/development. Contains a single demo user with hashed password credentials. Would be replaced by a real database table in production.
✅
app/validation.py
Sanitizes user input. Catches SQL injection patterns, strips XSS via Bleach, enforces message length limits. Runs before anything touches the database or the LLM.
Why: Defence-in-depth layer. Auth controls who gets in; validation controls what they can send. Together they're a complete defense.
⚡
app/cache_utils.py
TTL-based in-memory cache. Decorates RAG search with @cache_response(ttl=300). Avoids redundant ChromaDB queries.
Why: The 27% cost reduction came from caching + better context selection. This is half of that story.
📊
app/monitoring.py
Prometheus counters and histograms. Tracks requests, latency, token usage, bookings, active sessions. Exposed at /metrics.
Why: This is how you'd prove the system is healthy in production. Every metric here is an interview talking point.
📈
app/metrics_dashboard.py
Gradio-based web dashboard that fetches and displays Prometheus metrics from the /metrics endpoint. Refresh button for real-time monitoring. A lightweight alternative to full Grafana setup.
Why: Pairs with monitoring.py — one collects metrics, this one visualises them. Quick demo-friendly way to show observability without deploying Grafana.
🛡️
app/hallucination.py Quality
Scores every LLM response for hallucination risk. Three signals: token overlap (30%), semantic similarity via SentenceTransformer (50%), sentence-level source attribution (20%). Returns HIGH/MEDIUM/LOW confidence. Feeds Prometheus metrics.
Why: This is how you prove the AI isn't making things up. Every /chat response includes a confidence score from this module. Crucial for production trust.
📝
app/logging_utils.py
Structured JSON logging with request IDs. Makes debugging possible when things go wrong at 2am.
⏱️
app/profile_utils.py
@profile decorator that times function execution. Used on RAG search to measure latency.

The Features
PDF generation, streaming, the UI · 3 files · ~1 hr
▼
These are the "user-facing" pieces. The PDF generator creates booking confirmation documents, the stream client enables real-time chat, and the chat UI is the Gradio interface people actually interact with. All relatively self-contained.
📑
app/pdf_generator.py
Generates booking confirmation PDFs. Called by the generate_pdf tool when a guest wants their confirmation emailed/downloaded.
Why: Shows how tools can produce real outputs — not just text, but files. It's a complete feature end to end.
📡
app/stream_client.py
Client utility for streaming chat responses from the API using Server-Sent Events (SSE). Connects to /chat/stream and prints tokens in real-time as they arrive. Shows how to consume a streaming AI endpoint.
Why: Demonstrates the SSE streaming pattern — essential for ChatGPT-style "typing" UX. Short file (~18 lines), but the concept is interview gold.
💬
chat_ui.py
Gradio-based chat interface. Connects to the FastAPI backend. This is the thing you'd demo.
Why: Short and simple (~150 lines). Shows how to wrap an API in a UI fast. Good Gradio vs React talking point.

The ML Layer
Fine-tuning, evaluation, MLOps + benchmarking · 9 files · ~4 hrs
▼
This is the advanced ML engineering layer. LoRA fine-tuning, synthetic data generation, quality evaluation, MLflow experiment tracking, and RAG benchmarking. The MLflow tracker integrates into the main app for live metrics, while the experiment/benchmark scripts run as standalone workflows.
🧪
scripts/generate_synthetic_dataset.py ML
Calls GPT-4o to generate 150 Q&A pairs from templates + knowledge base. Outputs the training dataset.
Why: Shows how to create training data programmatically. The dataset it produces is what trains the LoRA model.
🎓
scripts/finetune_lora.py ML
LoRA fine-tuning on Phi-3-mini. Loads dataset, configures PEFT adapters, runs training loop. Only trains 0.5% of parameters.
Why: The centrepiece ML engineering file. Understanding this means you understand parameter-efficient fine-tuning.
🔬
scripts/test_lora.py ML
Tests the fine-tuned model. Compares outputs against the base model. Interactive and batch modes.
📈
notebooks/ragas_evaluation.ipynb ML
RAGAS framework evaluation. Measures faithfulness, answer relevancy, context precision. This is where the +26% accuracy number comes from.
Why: The metrics that prove the system works. Know these numbers cold for interviews.
📊
app/ragas_eval.py Eval
Python module for running RAGAS evaluation programmatically. Uses HuggingFace embeddings locally (no OpenAI dependency for evals). Computes faithfulness and answer relevancy on batch samples. Callable from scripts or tests.
Why: The notebook is for exploration; this module is for automation. Import it in CI/CD or benchmarking scripts to run quality checks without Jupyter.
⚖️
app/lora_integration.py ML
Plugs the fine-tuned model into the main app as a fallback when OpenAI is unavailable. Lazy-loads the model to save memory.
Why: The bridge between "trained a model" and "it's actually running in production."
📡
app/mlflow_tracking.py MLOps
MLflowTracker class — logs RAG evaluations, model configs, confidence metrics, and benchmark results to MLflow. Gracefully degrades to a no-op when MLflow is not installed or unreachable.
Why: This is how you track experiments and compare model/RAG configurations over time. Integrated into main.py for live confidence logging.
🧪
scripts/run_rag_experiment.py MLOps
Runs RAG evaluation experiments with different configurations and logs results to MLflow for comparison. Systematic way to test RAG parameter changes.
Why: Shows how to run reproducible ML experiments. Pair with MLflow UI (localhost:5000) to visualise results.
⚖️
scripts/benchmark_rag.py Benchmark
Head-to-head comparison of custom AdvancedRAG vs LangChain RAG. Ingests same data into both, runs identical queries, measures latency and quality. Logs results to MLflow.
Why: Proves why you built custom RAG. The numbers from this benchmark justify the architectural decision over using off-the-shelf LangChain.

The Deployment
AWS infrastructure + CI/CD · 7 files · ~1.5 hrs
▼
The cloud deployment layer. These files take the app from "running on my laptop" to "running on AWS with automated CI/CD." ECS Fargate for containers, ECR for images, Secrets Manager for credentials, CloudWatch for logs. Plus the FEATURES_GUIDE.md that documents everything.
📦
deploy/aws/ecs-task-definition.json AWS
ECS Fargate task definition. 1 vCPU, 2GB RAM, health checks, secrets from AWS Secrets Manager, CloudWatch logging. This is the blueprint for how your container runs in the cloud.
Why: Understanding task definitions is essential for any AWS container deployment. Every field maps to a production concern.
🚀
deploy/aws/ecr-push.sh AWS
Shell script that authenticates with ECR, builds the Docker image, tags it, and pushes it to your private container registry.
🔄
deploy/aws/deploy.sh AWS
Registers the task definition with ECS, updates the service to use the new image, and waits for stability. Zero-downtime rolling deployment.
📋
deploy/aws/cloudwatch-log-group.json AWS
CloudWatch log group config with 30-day retention. All container stdout/stderr goes here for debugging in production.
🐳
deploy/aws/docker-compose.prod.yml AWS
Production Docker Compose with awslogs driver, CPU/memory resource limits, and health checks. For EC2-based deployments as an alternative to Fargate.
⚙️
.github/workflows/ci.yml CI/CD
GitHub Actions workflow with test + deploy jobs. On push to main: lint, test, build Docker image, push to ECR, deploy to ECS. Full automated pipeline.
Why: This is the glue. Every git push triggers tests, and successful merges to main trigger production deployment.
📖
FEATURES_GUIDE.md Docs
Comprehensive documentation covering all 6 features — architecture diagrams, configuration tables, CLI commands, troubleshooting. The single reference for everything.
Why: Read this as a companion to the code. It explains the "why" behind every architectural choice and how to verify each feature.
💡 How to actually do this: One file per day. Read it, then open the app and try to find where that file gets called. Ask yourself "what breaks if I delete this?" That one question teaches you more than re-reading the code three times. The tests/ folder is your cheat sheet — the test names literally describe what each piece of the system does.
