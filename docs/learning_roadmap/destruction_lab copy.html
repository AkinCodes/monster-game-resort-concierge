System Overview & Dependency Graph

Architecture at a glance The app is a FastAPI server that acts as an AI concierge. A user sends a message to /chat. The message gets validated, the RAG system retrieves relevant knowledge, the LLM generates a response (possibly calling tools like book_room), and the response gets scored for hallucination confidence. Everything is logged to Prometheus and optionally MLflow. 
// Request Flow
User Message
  -> validate_message()           // validation.py
  -> jwt_or_api_key()             // auth_mixins.py (Depends)
  -> rag.search()                 // advanced_rag.py (BM25 + Dense + Rerank)
  -> router.chat()                // llm_providers.py (OpenAI -> Anthropic -> Ollama)
  -> registry.execute()           // tools.py (book_room, get_booking, search_amenities)
  -> router.chat()                // Phase 2: synthesis with tool results
  -> detector.score_response()    // hallucination.py (30% overlap + 50% semantic + 20% attribution)
  -> tracker.log_confidence()     // mlflow_tracking.py (optional)
  -> memory.add_message()         // memory.py -> database.py (SQLite)
  -> JSON Response

Purpose: The upgrade. Three-stage pipeline: (1) Hybrid retrieval — BM25 keyword search + dense embedding search, (2) Reciprocal Rank Fusion to merge both result sets, (3) BGE cross-encoder reranking for final ranking. This is what made the project go from "generic chatbot" to "actually accurate."

Interview questions this prepares you for
  • "Why hybrid search instead of just embeddings?" — Embeddings are great for semantic similarity but miss exact keyword matches. BM25 catches "Vampire Manor" exactly. RRF combines both strengths.
  • "Explain Reciprocal Rank Fusion." — score = sum(1/(k + rank)) for each result list. k=60 is standard. BM25 gets 40% weight, dense gets 60%. Documents appearing in both lists get boosted.
  • "What's a cross-encoder reranker?" — Unlike bi-encoders (encode query and doc separately), cross-encoders encode query+doc together. Slower but much more accurate. Used as final-stage filter on top-20 candidates.
  • "Why is BM25 in-memory only?" — Designed for long-running server process. BM25 is rebuilt on startup from ingested documents. ChromaDB persists to disk, BM25 doesn't. This is intentional — the server keeps it in RAM.

The Three-Stage Pipeline

# Stage 1: Retrieve candidates from both indices
bm25_results = self._bm25_search(query, k=20)     # keyword
dense_results = self._dense_search(query, k=20)    # semantic

# Stage 2: Merge with Reciprocal Rank Fusion
fused = self._reciprocal_rank_fusion(bm25_results, dense_results,
    k=60, bm25_weight=0.4)

# Stage 3: Rerank top candidates with cross-encoder
reranked = self._rerank(query, fused, top_k=5)

Interview questions this prepares you for
  • "How do you detect hallucinations?" — Three-signal scoring: (1) Token overlap between response and RAG contexts (30% weight), (2) Cosine similarity of embeddings (50%), (3) Sentence-level grounding — what fraction of response sentences have >30% overlap with context (20%).
  • "Why three signals instead of one?" — Each catches different things. Token overlap catches keyword matching. Semantic similarity catches paraphrasing. Source attribution catches fabricated sentences. Together they're robust.
  • "What are your thresholds?" — HIGH >= 0.7, MEDIUM >= 0.4, LOW < 0.4. Configurable via config.py.
  • "Why lazy-load the SentenceTransformer?" — It's ~90MB. Loading it on every import would slow down startup and waste memory if hallucination detection is disabled.

The Scoring Formula

overall = 0.3 * overlap + 0.5 * semantic + 0.2 * attribution

# Token overlap: what % of response words appear in context
overlap = len(response_tokens & context_tokens) / len(response_tokens)

# Semantic: cosine similarity of embeddings (max across contexts)
semantic = max(cosine_sim(response_emb, ctx_emb) for ctx in contexts)

# Attribution: fraction of sentences with >30% word overlap
attribution = grounded_sentences / total_sentences

  • "What is LoRA and why use it?" — LoRA freezes the base model and injects small trainable matrices into attention layers. Instead of fine-tuning 3.8B parameters, you train ~19M (0.5%). Same quality, fraction of the compute. Adapters are ~50MB vs 7GB for full model.
  • "Why Phi-3-mini?" — 3.8B params is small enough for local inference on a laptop. Phi-3 punches above its weight on reasoning tasks. Good trade-off between quality and deployability.
  • "What are target modules?" — q_proj, k_proj, v_proj, o_proj are the query, key, value, and output projection matrices in multi-head attention. These are where the model stores "what to pay attention to." Domain knowledge (Monster Resort specifics) is encoded here.
  • "What's 4-bit quantization?" — BitsAndBytes reduces weight precision from 32-bit float to 4-bit integer. Memory drops from ~15GB to ~4GB. Training uses QLoRA (quantized base + full-precision adapters). Minimal quality loss.
  • "Why only 0.5% of parameters?" — The base model already knows language. We only need to teach it Monster Resort domain knowledge. LoRA's low-rank matrices capture this efficiently without catastrophic forgetting.

notebooks/ragas_evaluation.ipynb + app/ragas_eval.py Deep Dive ▼

Purpose: RAGAS framework evaluation. Measures RAG quality across multiple dimensions. Compares three configurations: Basic RAG, Hybrid RAG, Hybrid + Reranking.

RAGAS Metrics (know these cold)

MetricWhat it measuresHow it works
FaithfulnessDoes the answer follow from the context?Decomposes answer into claims, checks each against context. Score = grounded claims / total claims.
Answer RelevancyDoes the answer address the question?Generates N questions from the answer, measures similarity to original question.
Context PrecisionIs the retrieved context relevant?Checks if retrieved chunks are relevant to the question. Higher = less noise.
Context RecallDoes the context cover the question?Checks if all parts of the reference answer can be found in retrieved context.
