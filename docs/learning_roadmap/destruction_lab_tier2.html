#18 Remove RRF BM25 Weight Reciprocal Rank Fusion NO TEST ▼
Hypothesis

Without BM25 scores in RRF, fusion becomes dense-only, losing keyword matching capability. Queries that depend on exact keyword matches will return significantly worse results.

Action

File: app/advanced_rag.py (lines 188–190)

Comment out the BM25 weight loop in the RRF calculation:

# for doc_id, bm25_score in bm25_results:
# rrf_scores[doc_id] += bm25_weight / (k + bm25_rank)
# bm25_rank += 1
Verify

Run a keyword-heavy query and observe degraded results:

curl -X POST http://localhost:8000/chat \
-H "Content-Type: application/json" \
-d '{"message": "Tell me about Vampire Manor"}'

Keyword queries like "Vampire Manor" return worse results because BM25 exact-match signal is missing.

Why It Matters

RRF combines keyword (BM25) and semantic (dense embedding) signals. Removing one leg of the fusion demonstrates how hybrid search outperforms either signal alone. Keywords handle proper nouns and exact terms; embeddings handle synonyms and concepts.

Interview Prep
  • What is Reciprocal Rank Fusion and how does it work?
  • Why combine keyword and semantic search instead of using one?
  • What types of queries does BM25 handle better than dense retrieval?
Recovery
