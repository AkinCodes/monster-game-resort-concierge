The Toolbelt — 3 Pouches on the Castle Wall

The belt hangs on the dungeon wall. Each pouch is a Tool dataclass: frozen, named, self-contained. Click any pouch to open it and see what's inside — the ingredients it needs, its personality, and the hotel checklist stitched into Pouch 1.

Think of it like a utility belt that Batman wears — one belt, multiple compartments. Each pouch holds a different tool.

The Instruction Manual

  

    Imagine you hire a new waiter at your restaurant. Smart guy. Speaks 12 languages. But he's never seen your menu.
  

  

    You can't just say "go take orders." You have to hand him the menu first — what's available, what comes with what, and what the customer HAS to choose (no skipping the main course!).
  

  

    That's exactly what the instruction manual is. The Oracle (LLM) is the smart waiter. The JSON schema is the menu. Without it, the Oracle doesn't even know the tools exist.
  


  

    🍴 The Sesame Street Version:


    The Oracle (the LLM) just arrived at the Monster Resort. Big Bird hands the Oracle the guest handbook he compiled from all the employee badges.


    Oracle: "Okay! But... what exactly can each employee do? What do I ask the guest?"


    Big Bird flips through the handbook:


    📋 BOOK A ROOM — You MUST ask: guest name, hotel name (ONLY from this list!), room type, check-in date, check-out date. No exceptions.


    📋 LOOK UP A BOOKING — You MUST ask: booking ID. That's it. Just the ID.


    📋 SEARCH AMENITIES — You MUST ask: what are you looking for? Just the question.


    Now the Oracle knows EXACTLY what to ask for. It doesn't guess. It doesn't improvise. It follows the handbook.


    👉 That handbook? That's the OpenAI function schema. Click each one below to read it.

    How it's called:
    registry.get_openai_tool_schemas()
    loops self.tools.values(), calls tool.to_openai_schema() on each,
    wraps the result in {"type": "function", "function": ...}, and returns a list.
    That list is passed to the OpenAI API as the tools parameter.
  


  

    🍪 Sesame Street Version:


    Remember Big Bird's handbook with 3 pages? Well, before the hotel opened, Big Bird had to actually MAKE those pages.


    Here's how he did it:


    Big Bird opened the staff directory and went down the list. One by one, he asked each employee:


    👉 "What's your name?" — "book_room!"

    👉 "What do you need from the guest?" — "guest_name, hotel_name, room_type, check_in, check_out!"

    👉 "Any rules?" — "hotel_name MUST be from the approved list!"


    Big Bird wrote all of this down on one handbook page. Then he did the same for the next employee. And the next.


    When he was done, he had 3 handbook pages — one per employee. He stacked them together and handed the whole handbook to the Oracle.


    "Here. This is everything my team can do. Don't lose it."


    Oracle: "I will guard it with my life!"


    👉 That's get_openai_tool_schemas(). It loops through every tool, asks each one to describe itself, wraps it in a nice format, and hands the whole stack to the Oracle. The Oracle reads the handbook and now knows exactly what tools exist and how to use them.

Sesame Street

          The hotel is open! A guest walks in. Big Bird is now the front desk dispatcher. ⏱


          Every time a guest makes a request, Big Bird does the EXACT same routine:


          1. 📋 "What's your request number?" (generates a UUID if there isn't one)

          2. 🔎 "Is that employee in the directory?" (checks the staff directory. If not: "ERROR! NO SUCH EMPLOYEE!")

          3. 🧹 Cleans up the request (some guests have weird colons stuck to their words — Big Bird strips those off)

          4. ⏱ STARTS THE STOPWATCH

          5. 📊 Makes a tick mark on the tally board in the lobby

          6. 🚀 Calls the right employee and WAITS

          7. ⏱ STOPS THE STOPWATCH — "That took 47 milliseconds!"

          8. 📝 Writes everything in the log: who ran, what happened, how long it took



          But what if the employee TRIPS AND FALLS? (an exception)


          Big Bird doesn't panic. He calmly:

          ⏱ Stops the stopwatch anyway

          📝 Writes "Something went wrong" in the log

          📦 Returns a polite message to the guest: "ok: false, error: they fell over"


          The hotel doesn't crash. The guest gets a polite message. Big Bird keeps dispatching.


          👉 That's async_execute_with_timing(). The reliable wrapper that times everything, logs everything, catches every error, and never lets the system crash.
        

        

          Core execution engine. Validates tool name, strips trailing colons from kwargs
          (a quirk of some LLM outputs), records start = time(), increments the
          Prometheus counter, awaits the tool function, calculates elapsed_ms,
          and logs success or failure. Returns the result or an error dict.
        

        returns: Any (result) | {"ok": false, "error": ...}
        

          async/await
          TOOL_CALL_COUNT.inc()
          elapsed_ms
          try/except

          He has three pieces of equipment in his arms:

          🗃 The filing cabinet (db — for storing bookings)

          🖨 The stamp machine (pdf — for printing receipts)

          Big Bird walks into the empty lobby and sets up three workstations:


          🏷 Station 1 (book_room): He puts the filing cabinet AND the stamp machine here. "This station needs BOTH to do its job."

          🏷 Station 2 (get_booking): He puts JUST the filing cabinet. "This one only needs to look things up."

          He puts laminated badges on each station, writes them all in the staff directory, and says:

          This happens ONCE, at opening time. The stations stay set up for the entire day (the entire server lifetime). Big Bird never has to set up again.


          👉 That's make_registry(db, pdf, rag). The morning routine that wires the right dependencies to the right tools, once, at startup.
        

        

          Module-level factory function. Receives the three dependencies as arguments, creates a
          fresh ToolRegistry, registers all three tool functions (which close over
          the dependencies via Python closures), and returns the finished registry. Called once
          at app startup — the result is reused for the entire server lifetime.
        

        returns: ToolRegistry (3 tools registered)
