The Hotel Filing System Analogy

Concept Mapping

Every component of the memory system has an exact operational counterpart inside Monster Resort.

Code Concept Hotel Equivalent Why it maps
Raw messages[ ] Individual guest request slips Each slip captures one moment: "Room 304 wants extra towels at 11:42pm." Unfiltered, unprocessed, exactly as spoken.
summary str Shift handover report / Guest profile card At shift change, the outgoing manager doesn't hand over 400 slips — they hand over a one-page profile: "Guest prefers room temp 19C, allergic to shellfish, complained about noise twice."
_maybe_summarise() The night manager compiling reports The night manager checks: is the filing cabinet getting full? If yes, they spend 10 minutes writing a summary before going off-duty. If not, they leave it for the next shift.
_cheap_summary() The trainee covering when the manager is sick No time, no LLM budget — the trainee just staples the last 5 slips together and writes "SEE ATTACHED" on the cover. Fast, ugly, but better than nothing.

RAM (in-memory dict) The front desk notepad Lightning fast to read and write — but the moment the front desk closes (server restart), the notepad is thrown away. It's working memory, not archive.

A Busy Night at the Front Desk

The day shift ends. Front desk manager Viktor Frankenbottom has handled 340 guests since sunrise. The filing cabinet behind him — labelled SESSION: LOBBY_MAIN — now holds 19 request slips, and the cabinet only holds 20.

GUEST [18:02] "I need extra pillows — the ones without feathers."
GUEST [19:15] "Can you recommend a restaurant near the swamp exhibit?"
GUEST [21:30] "The wifi password isn't working in room 208."
GUEST [23:44] "What time is the monster parade tomorrow?"

At 11:47 PM, a new guest approaches. Viktor reaches for the cabinet to file the 20th slip — and then the 21st request comes in simultaneously. The cabinet is full.

CABINET OVERFLOW — max_messages_before_summary = 20 reached.
Filing the 21st request without consolidating risks losing the oldest entries.

Viktor calls _maybe_summarise(). He reviews all 20 slips, finds the patterns, and writes a single Guest Profile Card:

SHIFT HANDOVER SUMMARY — Session LOBBY_MAIN
Guest has feather allergy (no feather pillows). Prefers restaurant "Swamp Bistro" (rec'd). Wifi issue in room 208 resolved at 22:15. Interested in tomorrow's 10am parade. No outstanding complaints. Tone: satisfied.

Viktor discards the 20 original slips and replaces them with this single card. The cabinet now has room for 20 fresh requests. This is rolling compression — the old raw data is gone, but the meaning survives.

What if the AI service that writes summaries is temporarily offline? Viktor calls _cheap_summary() — the emergency protocol. His trainee grabs the last 5 slips, clips them together, and stamps the top: "MOST RECENT INTERACTIONS — condensed by trainee." It's not elegant, but the cabinet stays functional. The LLM token budget was zero; at least nothing was lost.

At 6 AM, the day shift arrives. The night manager doesn't hand them 300 raw slips — they hand them the DB records (the basement archive of every stay since 2019) and the current summary card (last night's desk notepad). Fast handover, full context, zero token overflow.

02
Architecture Decision

DB vs RAM — Comparison Table

Dimension RAM Only (in-memory dict) Database (SQLite / Postgres)
Persistence Gone on server restart. Session history dies with the process. Survives restarts, crashes, deploys. Guest returns next week — history intact.
Multi-User Fine for single-process, single-worker. Breaks when you add workers (Gunicorn multi-process). All workers read/write the same DB. No data isolation bugs.
Crash Recovery Total loss. The notepad burns with the building. Transactions + WAL journaling. Even a mid-write crash recovers cleanly.
Search Must iterate entire dict. O(n) scan, no indexing. "Find all sessions mentioning hotels" = impossible at scale. SQL WHERE, LIKE, full-text search, indexes. O(log n) lookups.
Scaling Tied to one machine. Can't share state across pods in Kubernetes. External DB (Postgres, Redis) is accessible from any instance.
Token Management You still need rolling summarization — RAM doesn't help with context window limits. Same requirement. DB stores the summaries; it doesn't change the LLM's context limit.
Complexity 3 lines of code. dict[session_id] = messages. Zero setup. Schema design, migrations, connection pooling, cursor management. Real engineering.
Speed (read/write) Nanoseconds. In-process memory access, no I/O. Milliseconds (local SQLite) to tens of milliseconds (remote Postgres). Fast enough for chat.
Cost Free. Uses existing process memory. SQLite: free. Postgres: $0 self-hosted, $15–200/mo managed cloud. Worth it at scale.
Best For Demos, local dev, single-user scripts, Jupyter notebooks. Any production system with real users, multi-session history, or restart requirements.
Rule of thumb: Start with RAM when exploring. Switch to DB the moment you need history to survive a Ctrl+C, or the moment you have more than one user.

Database Concepts

fetchone() Analogies

🃏

The Card Deck

Imagine a database query returns a result set like a shuffled deck of cards, face-down on the table. The cursor is your hand hovering over the deck.

fetchone() picks the top card and shows it to you. The deck now has one fewer card, and your hand is positioned over the next. Call it again — another card. The deck has a position marker that advances with every pick.

When you call fetchall(), you pick up the entire remaining deck at once. The cursor remembers where you stopped — you can pick up mid-deck without redealing.

cursor → your hand above the deck
fetchone() → pick one card from the top
None returned → deck is empty
fetchall() → grab the rest of the deck
⚙️

The Conveyor Belt

Picture a sushi conveyor belt. The kitchen (database) has already prepared all the plates — they're lined up on the belt in order. The cursor is the person sitting at the counter.

fetchone() grabs the next plate as it arrives at your seat. You eat it, put the plate down, and the belt advances one position. You never need to know how many plates are coming — you just keep grabbing until the belt is empty.

This is memory-efficient: you never have all the plates stacked in front of you at once (unlike fetchall()). For 10,000 rows, fetchone() uses the same RAM as fetching 1 row.

kitchen prep → SQL query execution
belt position → cursor position in result set
grab one plate → fetchone()
empty belt → fetchone() returns None
🛎️

The Monster Resort Checkout Queue

It's Sunday morning checkout at Monster Resort. Fifty guests are lined up at the front desk. The cursor is the concierge's current focus — the guest they're actively processing.

fetchone() is the concierge saying: "Next please." They handle that one guest, complete the interaction, and call the next. They do not invite all 50 guests behind the desk simultaneously.

If the concierge steps away mid-queue (cursor not closed, connection kept open), the queue stays exactly as it was. They return, say "Next please," and continue from exactly where they left off. Close the connection and the queue disperses — you'd have to re-run the query to rebuild it.

checkout queue → query result set
"Next please" → fetchone()
concierge focus → cursor position
empty lobby → fetchone() returns None
05
Failure Modes

The "When Does It Break" Game

Scenario 1

User chats for 2 hours → Token overflow

What breaks

After 120 minutes of conversation, the raw message list has 240 turns. Each turn averages 80 tokens. That's 19,200 tokens — over the 16k context window of many models. The API call throws a ContextLengthExceededError. The bot stops responding entirely. Worse: earlier conversation turns get silently truncated, so the model "forgets" what the user said in minute 3 while still billing for the full context attempt.

The Fix

Implement _maybe_summarise() with a max_messages_before_summary threshold (e.g., 20 messages). Every time the list hits the threshold, compress all messages into a rolling summary. The context window stays bounded at roughly len(summary_tokens) + threshold_messages — never grows unbounded.

Scenario 2

Server restarts mid-conversation → Memory gone

What breaks

You store sessions in a Python dict: sessions = {}. Your app crashes at 3 AM (OOM kill, deploy, Heroku dyno restart). All conversations in RAM vanish. Users return the next morning and the bot has no memory of them. For a hotel concierge, this means a guest says "as I mentioned earlier, I have a feather allergy" and the bot replies "I don't have any record of that" — a trust-destroying moment.

The Fix

Write every message and every generated summary to the DB immediately after it's created (not batched, not deferred). On startup, re-hydrate the session from DB. Use the pattern: get_or_create_session(session_id) — look in DB first, fall back to fresh session only if truly new. The RAM dict becomes a write-through cache, not the source of truth.

Scenario 3

User asks "what did I say about hotels?" → Can't search RAM efficiently

What breaks

The user has 800 historical messages across 30 sessions stored in RAM as nested lists. Their question requires scanning every message for keyword relevance. In Python, that's a nested loop over ~24,000 string comparisons. No ranking, no relevance scoring. You either return the wrong result, time out, or load all 800 messages into the LLM context and blow the token budget just to answer a simple retrieval question.

The Fix

Store messages in a DB with full-text search or use a vector store for semantic search. SQLite supports FTS5 (full-text search). For semantic search, embed messages with a small model and query by cosine similarity. The bot can then answer "what did you say about hotels?" by retrieving the 3 most relevant message chunks rather than loading your entire history.

Scenario 4

Two users with same session_id → Data collision

What breaks

Your session IDs are generated as f"session_{user_id}" but two users share the same account login. Or worse — your ID generation has a collision bug and two strangers land in the same session bucket. User A is a vegan asking about the swamp buffet menu. User B is asking about room rates. The bot merges their context and responds to User A with room pricing — or worse, shares User B's personal preferences with User A. This is a GDPR nightmare.

The Fix

Use cryptographically random session IDs: import secrets; session_id = secrets.token_hex(32). Add a DB constraint: UNIQUE(session_id). In multi-user systems, always scope sessions to (user_id, session_id) pairs — never allow one session to bleed into another. Audit your session isolation in tests with two concurrent users before any production deploy.

Scenario 5

Summary grows to 10,000 tokens → Defeats the purpose

What breaks

Your prompt to the LLM asks it to summarize, but doesn't constrain the output length. After 50 compression rounds, each round adds 200 tokens to the summary instead of replacing them. The summary itself becomes a 10,000-token document. You're back to the original problem — the context window is still full — except now it's full of summaries of summaries, not raw messages. The compression loop has leaked.

The Fix

Always include a hard token cap in your summarization prompt: "Respond in no more than 300 words." Additionally, pass max_tokens=400 to the API call for the summary request itself — this enforces the limit at the API level, not just the prompt level. Monitor summary token length over time and alert if it exceeds a threshold. A well-designed rolling summary should stay roughly constant in length after the first few rounds.

Architecture Mental Model

The Tiered Memory Pyramid

Think of LLM memory as three layers with a fundamental tradeoff: speed vs. capacity vs. precision. The higher up the pyramid, the more immediate but the smaller the space.
Tier 1 — Active

The Context Window

What lives here: Everything the LLM can see right now during a single API call. The system prompt, the rolling summary, and the most recent N raw messages.

  • Capacity: 4k–128k tokens depending on model
  • Duration: One inference call only
  • Access: Instant — it's literally the input to the model
  • Monster Resort equivalent: What Viktor is actively reading at the front desk right now
⚡ Nanosecond access 📏 Limited capacity
↓
Tier 2 — Compressed

Rolling Summaries

What lives here: The distilled essence of all prior conversation rounds — what the LLM decided was worth keeping after compressing raw messages. Stored in DB, loaded into Tier 1 at the start of each call.

  • Capacity: Bounded by your max_summary_tokens setting (~300–800 tokens typically)
  • Duration: Persists across calls, survives restarts (if in DB)
  • Access: One DB read on session load, then lives in Tier 1
  • Tradeoff: Loses precise wording and timestamps; keeps semantic meaning
  • Monster Resort equivalent: The shift handover card — Viktor reads it once at the start of his shift
💾 DB-backed 🦹 Lossy compression 🔄 Updated each summarization cycle
↓
Tier 3 — Archived

Raw Message Archive

What lives here: Every original message, verbatim, with timestamps and metadata. Not loaded into the LLM by default — retrieved only when explicitly needed (search, audit, replay).

  • Capacity: Unlimited (disk storage, DB rows)
  • Duration: Permanent, unless you have a data retention policy
  • Access: SQL query + optional vector search — milliseconds to seconds depending on scale
  • Use cases: "What exactly did the user say at 2:34 PM?" / GDPR data export / debugging / model fine-tuning datasets
  • Optional: many production bots skip this tier entirely and only keep summaries
  • Monster Resort equivalent: The fireproof basement archive — nobody reads it daily, but it's there when needed for legal, audit, or a returning guest requesting their history
🗃️ Unlimited storage 🔍 Searchable ⏱️ Slow retrieval ⚠️ Optional

How Data Flows Through the Pyramid

Downward (write path): New messages enter at Tier 1 as raw inputs. When the threshold is hit, _maybe_summarise() compresses them: raw messages move to Tier 3 archive (optional), the new summary overwrites Tier 2 in the DB, and Tier 1 gets cleared to just the fresh summary + recent messages.

Upward (read path): On session load, the system reads the Tier 2 summary from DB and loads it into Tier 1. If the user asks a specific historical question ("what did I say about my allergy in January?"), the app queries Tier 3 by keyword or vector similarity and injects the retrieved snippet into Tier 1 context.

The golden rule: The LLM only ever directly sees Tier 1. Tiers 2 and 3 exist to feed Tier 1 efficiently without overwhelming it. A well-designed system keeps Tier 1 below 30% capacity during normal operation, leaving headroom for the user's next message and the model's response.

Interview Preparation

Scenario-Based Interview Questions

Question 1 — Debugging

"Your chatbot works fine for 5 minutes but breaks after 30 minutes of conversation. Why?"

Ideal Answer
Senior-Level Response

The classic symptom points to unbounded context growth. In the first 5 minutes, the message list is small — say 20 messages — well within the model's context window. After 30 minutes of active chatting (assume 4 turns/minute), you have ~240 messages. At ~80 tokens per message, that's 19,200 tokens. Most models have 4k–16k context limits — you've overflowed.

I'd diagnose it by checking three things in order:

  • 1. Log the token count before each API call. You'll see it grow linearly until the call fails.
  • 2. Check the error type. ContextLengthExceededError or HTTP 400 from the API confirms the hypothesis.
  • 3. Inspect the message list length in your session object at the time of failure.

The fix is rolling summarization: implement a max_messages_before_summary threshold. When messages exceed it, compress them into a summary via an LLM call (or _cheap_summary() as fallback), discard the raw messages, and continue. The context window stays bounded regardless of how long the conversation runs.

Bonus: mention that simply truncating old messages (the naive fix) loses important early context — for example, the user's stated dietary restriction from minute 2 would vanish. Summarization preserves meaning while reducing tokens.

Question 2 — System Design

"How would you design a memory system for a chatbot that needs to remember conversations across days?"

Ideal Answer
Senior-Level Response

I'd build a three-tier system: active context window (what the LLM sees now), persistent compressed summary (what's been condensed), and optional raw archive (verbatim history).

Session persistence: Every message written to DB immediately — never RAM-only. Use session_id (UUID) as the primary key. On any new chat request, load the session summary from DB into the context window before constructing the prompt.

Cross-session memory: Store a user-level persistent profile (name, preferences, allergy flags, VIP status) separate from individual session summaries. On session load, inject the user profile first, then the session summary, then recent messages.

Summarization strategy: Rolling compression triggered at a message threshold within a session. At session close (or after a period of inactivity), run a "session close summary" that distills the entire session into 2–3 sentences and merges it into the user profile.

Trade-offs I'd discuss:

  • LLM-based summarization vs. extractive keyword summarization — LLM is higher quality but slower and costs money.
  • Storage cost of raw archives vs. convenience for debugging and GDPR compliance.
  • How many days of history to include in context vs. recency bias — a conversation from 60 days ago may be irrelevant and should be deprioritized.
Question 3 — Engineering Tradeoffs

"What's the tradeoff between using an LLM to summarize vs. regex extraction?"

Ideal Answer
Senior-Level Response

These are fundamentally different tools addressing different problems. The choice depends on what you need to preserve and how much you can spend.

LLM Summarization:

  • Preserves: Semantic meaning, tone, intent, inferred context ("the user seemed frustrated")
  • Loses: Exact wording, timestamps, structured data fields
  • Cost: API call per summarization cycle (~0.5–2 cents for GPT-4o-mini at 1k tokens)
  • Latency: 1–5 seconds for the summarization call
  • Failure mode: If the LLM API is down, summarization fails — use _cheap_summary() as fallback
  • Best for: Conversational context, emotional tone, implicit preferences

Regex / Rule-Based Extraction:

  • Preserves: Specific structured data exactly — phone numbers, dates, reservation codes, allergy flags
  • Loses: Nuance, implicit meaning, anything not matching the pattern
  • Cost: Zero — pure Python, no API call
  • Latency: Microseconds
  • Failure mode: Brittle — "I have a nut allergy" extracts; "tree nuts give me hives" may not
  • Best for: Contact info, structured fields, deterministic flags

The production answer is usually both: use regex to extract and store structured fields (allergy type, room number, booking ID) deterministically, and use LLM summarization to capture everything else. Regex gives you reliable, queryable structured data. LLM gives you the narrative context that would otherwise be lost. Neither alone is sufficient for a production hotel concierge system.
