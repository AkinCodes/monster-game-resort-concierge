#1 Remove BM25 Positive-Score Filter Search Quality Filtering ▾
Hypothesis

Without the >0 filter, BM25 will return documents with zero relevance scores, polluting the fusion step with noise.

Action

app/advanced_rag.py (lines 146-148)

Comment out:

return [(int(idx), float(scores[idx])) for idx in top_indices if scores[idx] > 0]

Replace with:

return [(int(idx), float(scores[idx])) for idx in top_indices]
Verify
uv run pytest tests/ -k "rag or search" -v

Expected: Search results include irrelevant documents with score 0.0

Why It Matters

Score thresholds are the first line of defense against noise in retrieval systems.

Interview Prep
  • How do you handle zero-relevance results?
  • Difference between BM25 and TF-IDF?
  • Why filter before fusion?
Recovery
git checkout -- app/advanced_rag.py

#2 Reverse Sort Direction in BM25 Sort Order in Ranking ▾
Hypothesis

Removing [::-1] reverses from descending to ascending, returning the LEAST relevant documents first.

Action

app/advanced_rag.py (line 145)

Comment out:

top_indices = np.argsort(scores)[::-1][:k]

Replace with:

top_indices = np.argsort(scores)[:k]
Verify
uv run pytest tests/ -k "search" -v

Expected: worst-matching docs returned first

Why It Matters

np.argsort returns ascending order by default. Forgetting to reverse is a subtle but devastating bug.

Interview Prep
  • What does np.argsort return?
  • Why is sort direction critical in ranking?
  • How would you detect inverted ranking in production?
