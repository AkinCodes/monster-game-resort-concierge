1. Project Overview

MCProject is a premium iOS concierge application for Soho House members. It provides two interaction paradigms: a Tool Picker mode for structured, form-based tool invocations and a Conversational Chat mode powered by OpenAI's GPT-4o, where users type natural language requests and the AI automatically determines which backend tools to call.

The app communicates with a Model Context Protocol (MCP) gateway server that exposes hospitality tools (venues, events, hotels, restaurants, bookings). In Chat mode, OpenAI acts as an orchestration layer, interpreting user intent, selecting the right MCP tool, formatting arguments, executing calls, and synthesising results into a natural language response with rich UI cards.

Key Technologies

Language: Swift 5.9+  |  UI: SwiftUI  |  Architecture: MVVM

AI Model: OpenAI GPT-4o (function calling)  |  Backend: MCP Gateway (JSON-RPC 2.0 over SSE)

Min iOS: 17.0  |  Xcode: 15+

2.2 MVVM Pattern

The app follows the MVVM (Model-View-ViewModel) pattern:

  • Model: MCPModels.swift defines Venue, Event, Hotel, Restaurant, and the generic MCPListResponse<T> wrapper.
  • View: ChatView.swift is the primary screen, containing mode toggle, message list, input areas, and reusable components (ModernTextField, PremiumButton, SuggestionChip, etc.).
  • ViewModel: ChatViewModel is an ObservableObject holding all published state. It orchestrates both modes and bridges between views and services.

The ChatView owns the ChatViewModel via @StateObject. Card sub-views receive the ViewModel as an @ObservedObject for interaction callbacks (e.g., tapping a venue card to fetch its open times).

4. Dual Mode System: Tools vs Chat

The app supports two interaction modes, controlled by a ChatMode enum and a segmented Picker in the header:

Tool Picker Mode .toolPicker

  • User selects a tool from a horizontal scroll picker
  • Structured form inputs appear (text fields, date pickers, steppers)
  • Arguments are explicitly constructed by the user
  • Direct MCP tool call via AgentService
  • Response decoded and displayed as rich cards
  • No AI summarisation - raw results only

Conversational Mode .conversational (Default)

  • Single text input field ("Ask me anything...")
  • Natural language processed by OpenAI GPT-4o
  • AI determines which tool(s) to call and with what arguments
  • MCP tools executed automatically via bridge function
  • Rich cards displayed + AI-generated text summary
  • Multi-turn conversation history maintained
// ChatMode enum definition
enum ChatMode: String, CaseIterable {
    case toolPicker = "Tools"
    case conversational = "Chat"
}

The mode toggle is a SwiftUI Picker with .segmented style, bound to viewModel.chatMode. The inputArea property conditionally renders either conversationalInputArea or toolPickerInputArea based on the current mode.

5.2 Function Calling Loop

When OpenAI determines that a tool call is needed, it returns a response with finish_reason: "tool_calls" instead of a text response. The service implements a loop that handles this iteratively:

OpenAI Function Calling Loop
1. User message appended to conversationHistory
2. LOOP (max 10 iterations):
   a. POST conversationHistory + toolSchemas to OpenAI
   b. Receive response
   c. IF finish_reason == "tool_calls":
      - Append assistant message (with tool_calls) to history
      - For each tool_call:
        i.   Parse tool name + arguments
        ii.  Execute via closure: executeTool(name, args)
        iii. Append tool result to history
      - Continue loop (go to step 2a)
   d. ELSE (finish_reason == "stop"):
      - Append assistant text to history
      - Return text to caller
3. If max iterations reached, return fallback message

Why a loop?

OpenAI may need to call multiple tools in sequence. For example, a booking request requires: (1) search restaurants to get venue IDs, (2) check availability, (3) confirm the table. Each step returns tool results that OpenAI uses to decide the next action.

5.4 System Prompt

The system prompt establishes the AI's persona and provides critical operational rules:

You are a premium Soho House member concierge assistant. You help members
discover venues, find events, check hotel availability, find restaurants,
and manage table bookings.

Today's date is {dynamically injected}. Use this to interpret relative
dates like "today", "tomorrow", "this week", "next Friday", etc.

IMPORTANT DATE FORMATTING RULES:
- For event queries (get_events): use YYYY-MM-DD format only
- For restaurant/hotel queries: use ISO 8601 with Z suffix
  (e.g. "2026-02-05T19:00:00Z"). NEVER use +00:00 offset.
- For hotel dates: use YYYY-MM-DD format

IMPORTANT BOOKING FLOW:
- Before checking table availability or confirming a table,
  ALWAYS search for restaurants first using search_specific_restaurants
  to get valid restaurant venue IDs.
- Use the exact venue IDs returned from search results.
- When confirming a table, you MUST ask the user for their mobile number.
- Always set confirmed_terms_and_conditions to true when the user
  wants to confirm.

NUMERIC PARAMETERS:
- party_size, number_of_rooms_stays, adults, children, limit must be
  whole integers (e.g. 2, not 2.0)

Dynamic Date Injection

The system prompt is a computed property that injects today's date at runtime. This allows GPT-4o to correctly interpret relative date expressions like "tonight", "this weekend", "next week", etc.

6.2 AgentService (AgentAPI)

The AgentService.swift file contains the AgentAPI class which handles all MCP communication. Key features include:

  • JSONValue enum: Lightweight type-safe JSON encoding for tool arguments (.string, .number, .bool, .array, .object, .null)
  • Dual contract support: RPC (default) and REST, with automatic fallback on 500 errors
  • SSE stream parsing: Reads event: message / data: {...} lines from the response
  • Tool name fallback mapping: Handles legacy tool name redirections
  • Debug logging: Generates curl commands for debugging failed requests
  • Error handling: Extracts trace IDs from error responses for server-side debugging

Core Method Signature

func callMCPTool(
    toolName: String,
    arguments: [String: JSONValue],
    chatSession: String,
    mcpURL: URL
) async -> AsyncStream<AgentPartialResponse>

The method returns an AsyncStream, allowing the caller to process partial responses as they arrive from the SSE stream. The stream yields AgentPartialResponse objects containing arrays of AgentMessage (currently supporting the .text case).

Step-by-Step Breakdown

  1. User input: Text captured in conversationalInput, sent via sendConversationalMessage()
  2. UI feedback: User message bubble + "Thinking..." indicator appended to messages
  3. OpenAI call: openAIService.sendMessage(text, executeTool: closure) sends to GPT-4o with all 11 tool schemas
  4. Tool detection: OpenAI returns finish_reason: "tool_calls" with get_specific_venues(location: "London", features_or_amenities: "pool")
  5. Bridge execution: The closure calls executeMCPToolForOpenAI(toolName:arguments:)
  6. Argument sanitization: sanitizeArgumentsForMCP() normalises dates and number types
  7. MCP call: aiService.callMCPTool() sends JSON-RPC 2.0 request to MCP gateway
  8. SSE response: Gateway streams back venue data as SSE events
  9. Rich cards: processToolResultForDisplay() decodes JSON into [Venue] and appends venue cards to messages
  10. Return to OpenAI: Raw JSON text returned to OpenAI as tool result
  11. Summary generation: OpenAI receives tool results, generates natural language summary
  12. Final display: "Thinking..." removed, AI summary text bubble appended

In Tool Picker mode, there is no OpenAI involvement. The ViewModel directly constructs arguments from form inputs and calls the MCP gateway. Responses are decoded through the same processMCPResponse() pipeline that attempts multiple JSON decoding strategies:

  1. Try MCPListResponse<Venue> (wrapped response with items/results key)
  2. Try MCPListResponse<Event>, <Hotel>, <Restaurant>
  3. Fallback to direct array: [Event], [Venue], [Hotel], [Restaurant]
  4. Final fallback: minimal event parsing from raw JSON dictionaries
  5. Last resort: display raw text

7.3 Multi-Turn Conversation

In conversational mode, OpenAIService maintains the full conversationHistory array. This enables contextual follow-up queries: 

// Conversation history structure:
[
    .system("You are a premium Soho House concierge..."),
    .user("Find venues in London with a pool"),
    .assistantWithToolCalls([get_specific_venues(location: "London", ...)]),
    .toolResult(toolCallId: "call_abc", content: "{venues JSON...}"),
    .assistant("I found 3 venues in London with pools: ..."),
    .user("What events are happening at those venues this week?"),    // <-- follow-up
    .assistantWithToolCalls([get_events(venue_ids: [...], ...)]),      // <-- AI has context!
    .toolResult(toolCallId: "call_def", content: "{events JSON...}"),
    .assistant("Here are the upcoming events at those venues: ...")
]

Context Awareness

Because the full conversation history is sent with each request, GPT-4o understands context like "those venues" and can reference venue IDs from previous tool results without the user needing to specify them again.

8. Data Models

ChatMessage (Display Model)

enum ChatMessage: Identifiable, Equatable {
    case text(id: UUID, content: String, isUser: Bool)
    case venueList(id: UUID, venues: [Venue])
    case eventList(id: UUID, events: [Event])
    case hotelList(id: UUID, hotels: [Hotel])
    case restaurantList(id: UUID, restaurants: [Restaurant])
}

This enum drives the message list UI. Each case renders differently in MessageBubble: text cases show bubbles, list cases show rich cards.

JSONValue (Argument Encoding)

enum JSONValue: Codable {
    case string(String)
    case number(Double)
    case bool(Bool)
    case array([JSONValue])
    case object([String: JSONValue])
    case null
}

Used throughout the app for MCP tool arguments. Provides type-safe JSON encoding without relying on Any or JSONSerialization. The OpenAI bridge converts from JSONSerialization output to JSONValue using the convertToJSONValue() helper, which includes special boolean detection via CFBooleanGetTypeID().

Summary

MCProject is a dual-mode iOS concierge app that combines:

  • Tool Picker Mode: Direct, form-driven MCP tool invocations for precise control
  • Conversational Mode: OpenAI GPT-4o function calling to orchestrate MCP tools via natural language

The OpenAI layer acts as an intelligent orchestration middleware, translating natural language into structured tool calls, executing them via the existing MCP infrastructure, and synthesising results into human-friendly summaries alongside rich UI cards. The argument sanitization layer bridges format differences between OpenAI's output and MCP's requirements, while multi-turn conversation history enables contextual follow-up queries.
