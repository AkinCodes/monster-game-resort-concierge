# Confluence: Monitoring Setup for Monster Resort Concierge (MRC)

## 1. Executive Summary

The Monster Resort Concierge (MRC) monitoring stack provides real-time visibility into the health and performance of our AI-driven hospitality services. By integrating Prometheus for data collection and Grafana for visualization, we ensure that both engineering and business stakeholders can track system stability and user engagement. This document serves as a comprehensive guide to the setup, troubleshooting, and interpretation of our monitoring metrics.

---

## 2. Architecture Overview

Our monitoring ecosystem consists of four main layers:

1. **The Application (Instrumented Code):** The MRC application exports internal performance data (like request counts and latency) in a format Prometheus understands.
2. **Prometheus (The Collector):** An open-source database that periodically "scrapes" (pulls) data from the application.
3. **Grafana (The Dashboard):** A visualization tool that queries Prometheus to build human-readable charts and alerts.
4. **MLflow (Experiment Tracking):** Tracks model experiments, RAG evaluation metrics, and hallucination confidence scores. Accessible at `http://localhost:5000`.

---

## 3. Key URLs and Access Points

For developers and operators, the following endpoints are essential for verifying the flow of data:

### **A. Application Metrics Endpoint**

* **URL:** `http://api:8000/metrics` (internal) or `http://localhost:8000/metrics` (local)
* **What it is:** A raw text page generated by the application.
* **Why it's necessary:** This is the "source of truth." If a metric doesn't appear here, Prometheus cannot record it. It shows technical data like `mrc_http_requests_total` and `process_cpu_seconds_total`.

### **B. Prometheus Web UI**

* **URL:** `http://localhost:9090`
* **What it is:** The management console for our data collector.
* **Why it's necessary:** Used to verify that the application is "UP" and to run raw queries (PromQL) to troubleshoot data gaps before building dashboards.

### **C. Grafana Dashboards**

* **URL:** `http://localhost:3000`
* **What it is:** The primary interface for non-developers and stakeholders.
* **Why it's necessary:** It converts complex numbers into visual trends, such as user session growth or error rate spikes.

---

## 4. Setup and Verification Steps

### **Step 1: Verify Application Connectivity**

Before configuring dashboards, ensure Prometheus can "see" the application.

1. Navigate to the Prometheus UI.
2. Go to **Status -> Target health**.
3. Look for the `monster-resort-concierge` job. It should show a state of **UP**.
* *Non-Developer Note:* "UP" means the two systems are successfully talking to each other.

### **Step 2: Identifying the Correct Metrics**

A common point of confusion is using standard names like `http_requests_total`. In our system, metrics are prefixed with **`mrc_`** to distinguish our custom business logic from standard system metrics.

* **Correct Name:** `mrc_http_requests_total`
* **Incorrect Name:** `http_requests_total`

### **Step 3: Creating Visualizations**

When building a panel in Grafana, we use the **PromQL** language. To see the "Speed" of traffic rather than just the "Total," we use the `rate()` function.

---

## 5. Technical Deep Dive: Why "No Data" Happens

One of the most frequent issues encountered is a "No Data" message in Grafana, even when the system is running.

### **The Role of the `rate()` Function**

The `rate()` function calculates how much a counter (like total requests) has increased over a specific window of time.

* **The Trap:** If you set a window that is too small (e.g., `rate(...[10s])`) and your system only scrapes data every 15 seconds, Prometheus won't have enough data points to do the math.
* **The Fix:** Always use a window that covers at least two or three scrape intervals. We recommend using `[$__rate_interval]` in Grafana, which adjusts automatically.

## 6. Understanding the Metrics (The "What" and "Why")

| Metric Name | Type | Audience | Why it matters |
| --- | --- | --- | --- |
| `up` | Gauge | DevOps | Tells us if the application is crashed or reachable. |
| `mrc_http_requests_total` | Counter | Developers | Tracks every hit to the API. Helps identify which features are most popular. |
| `mrc_active_sessions` | Gauge | Product Managers | Shows how many users are currently interacting with the concierge. |
| `mrc_ai_tokens_total` | Counter | Finance/Business | Directly correlates to the cost of running our AI models. |
| `mrc_http_request_latency_seconds` | Histogram | Engineers | Measures "slowness." Essential for ensuring the AI doesn't keep guests waiting. |

---

## 7. Troubleshooting FAQ

**Q: I see a jagged line in my graph that keeps dropping to zero.**

* **Cause:** You are likely plotting a raw counter. Because counters only go up, any minor delay in scraping can look like a "drop."
* **Solution:** Wrap the metric in `rate(...)`. This smooths the data into a per-second average.

**Q: Prometheus says "Empty query result" for my metric.**

* **Cause:** This is usually a typo or a namespace issue.
* **Solution:** Check the `/metrics` endpoint directly to confirm the exact spelling of the metric.

**Q: Why do we care about `python_gc_objects_collected_total`?**

* **Explanation:** While not a "business" metric, this tracks Python's memory management (Garbage Collection). If this number spikes, it might explain why the app is feeling sluggish.

---

## 8. Summary for Stakeholders

Monitoring is not just about catching errors; it is about understanding our guests' journey. By tracking `mrc_bookings_total` alongside `mrc_ai_tokens_total`, we can calculate the "Cost per Booking," allowing the business to make data-driven decisions on AI scaling.

### **Confidence & Quality Metrics**

The `/chat` endpoint now returns a `confidence` field (0.0-1.0) produced by the hallucination detection system (`app/hallucination.py`). This score indicates how grounded the response is in the knowledge base context. MLflow (accessible at `http://localhost:5000`) logs these confidence scores per experiment run, enabling trend analysis over time. The docker-compose stack now includes four services: `api`, `prometheus`, `grafana`, and `mlflow`.

For technical assistance or access to the Grafana dashboards, please contact the DevOps team.
