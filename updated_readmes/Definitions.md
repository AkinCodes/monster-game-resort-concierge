# Definitions.md

## Key Terms and Definitions

This document provides definitions for important terms and concepts used in the Monster Resort Concierge project.

---

### RAG (Retrieval-Augmented Generation)
A system that combines information retrieval (searching documents or databases) with generative AI models to produce more accurate and grounded responses.

---

### Tool Call
A mechanism where the language model can invoke external functions ("tools") such as booking a room, searching knowledge, or generating invoices, and use their results in its responses.

---

### Session ID
A unique identifier used to track a user's conversation and context across multiple API requests.

---

### Vector Store
A database optimized for storing and searching high-dimensional vectors, typically used for semantic search in RAG systems (e.g., ChromaDB).

---

### LoRA (Low-Rank Adaptation)
A parameter-efficient fine-tuning technique for large language models, allowing quick adaptation to new tasks or domains with minimal compute.

---

### API Key
A secret token used to authenticate and authorize requests to the backend API.

---

### Assistant Message
A message generated by the AI assistant, typically in response to a user query, which may include tool call results or direct answers.

---

### ToolRegistry
A component that manages the registration, listing, and invocation of available tools for the agent or LLM.

---

### ModelRouter
A multi-provider LLM orchestration component (`app/llm_providers.py`) that manages an automatic fallback chain: OpenAI -> Anthropic (Claude) -> Ollama (local). If the primary provider fails or is rate-limited, the router seamlessly tries the next provider in the chain. Configured via the `MRC_MODEL_PROVIDER` environment variable.

---

### HallucinationDetector
A response validation component (`app/hallucination.py`) that scores every `/chat` response by cross-referencing the LLM output against retrieved RAG context. Returns a ConfidenceLevel (HIGH, MEDIUM, or LOW) and a numeric score. Helps ensure the concierge only provides information grounded in the knowledge base.

---

### ConfidenceLevel
An enum used by the HallucinationDetector to classify response trustworthiness:
- **HIGH** (score >= 0.7): Response is well-grounded in retrieved context.
- **MEDIUM** (score >= 0.4): Response is partially grounded; review may be needed.
- **LOW** (score < 0.4): Response may contain hallucinated information.

---

### MLflow
An open-source MLOps platform (`app/mlflow_tracking.py`) used for experiment tracking, model versioning, and metric logging. Accessible at http://localhost:5000 when running via docker-compose. Tracks RAG evaluation metrics, model performance, and training runs.

---

### Centralized Config
The `app/config.py` module that consolidates all project configuration into a single location. All environment variables use the `MRC_` prefix (e.g., `MRC_MODEL_PROVIDER`, `MRC_API_KEY`, `MRC_MLFLOW_TRACKING_URI`). This replaces scattered `os.getenv()` calls throughout the codebase.

### **14. Hybrid Search**
**Definition:** Combining multiple search methods (typically BM25 + semantic) to get benefits of both keyword matching and conceptual understanding.

**In Your Project:**
```python
# app/advanced_rag.py
def search(self, query: str, k: int = 5):
    # Stage 1: Run both searches in parallel
    bm25_results = self._bm25_search(query, k=20)     # Keyword
    dense_results = self._dense_search(query, k=20)   # Semantic
    
    # Stage 2: Combine using Reciprocal Rank Fusion
    fused = self._reciprocal_rank_fusion(bm25_results, dense_results)
    
    # Stage 3: Rerank with cross-encoder
    final = self._rerank(query, fused, top_k=k)
    return final
```

**Why Both?**
```
Query: "Tell me about Vampire Manor"

BM25 (Keyword):
  ✅ Finds exact "Vampire Manor" mentions
  ❌ Misses "nocturnal lodging" synonyms

Dense (Semantic):
  ✅ Finds vampire-related concepts
  ❌ Might rank generic "manor" docs high

Hybrid:
  ✅ Gets exact matches from BM25
  ✅ Gets semantic matches from dense
  ✅ Fusion picks the best of both
  
Result: 100% accuracy on proper nouns!
```

### **15. Reciprocal Rank Fusion (RRF)**
**Definition:** A method for combining ranked lists from multiple sources. Gives higher weight to items that rank well in multiple lists.

**Formula:**
```python
RRF_score(doc) = sum(1 / (k + rank_in_list_i))
# k = constant (usually 60)
```

**In Your Project:**
```python
def _reciprocal_rank_fusion(self, bm25_results, dense_results):
    # BM25 ranks:  DocA=1, DocB=2, DocC=10
    # Dense ranks: DocB=1, DocA=3, DocD=5
    
    # RRF scores:
    # DocA: 1/(60+1) + 1/(60+3) = 0.0164 + 0.0159 = 0.0323
    # DocB: 1/(60+2) + 1/(60+1) = 0.0161 + 0.0164 = 0.0325 ← Highest!
    # DocC: 1/(60+10) + 0 = 0.0143
    # DocD: 0 + 1/(60+5) = 0.0154
    
    # Final order: DocB, DocA, DocD, DocC
```

**Why It Works:** Documents that appear in both lists get boosted, ensuring consensus.

### **16. Reranking (Cross-Encoder)**
**Definition:** A second-stage model that scores query-document pairs together for final precision. More accurate but slower than initial retrieval.

**In Your Project:** Using BGE cross-encoder (`BAAI/bge-reranker-base`).

**Two-Stage Architecture:**
```
Stage 1 (Fast, less accurate):
  BM25 + Dense Embeddings → 20 candidates (100ms)

Stage 2 (Slow, very accurate):
  Cross-Encoder → Re-score top 20 → Pick top 5 (300ms)

Total: 400ms (vs 5000ms if cross-encoder scored all docs)
```

**How Cross-Encoder Works:**
```python
# Bi-encoder (semantic search):
query_emb = encode("spa services")       # [0.2, 0.5, ...]
doc_emb = encode("Lightning Spa")        # [0.3, 0.4, ...]
score = cosine(query_emb, doc_emb)       # 0.85

# Cross-encoder (reranker):
score = cross_encoder.predict([
    ("spa services", "Lightning Spa")    # Encodes TOGETHER
])  # 0.94 (more accurate!)
```

**Improvement:** Reranking boosted your context precision from 0.85 to 0.94 (+11%).

### **36. LoRA (Low-Rank Adaptation)**
**Definition:** Parameter-efficient fine-tuning technique. Instead of updating all model weights, trains small adapter layers. Reduces training cost by 90%.

**In Your Project:**
```python
# Full fine-tuning:
Parameters to train: 3,800,000,000 (3.8B) ❌ Expensive!
VRAM needed: 32GB
Training time: 200 hours on CPU

# LoRA fine-tuning:
Parameters to train: 20,000,000 (20M) ✅ 0.5% of total!
VRAM needed: 8GB
Training time: 8-12 hours on CPU
```

**How LoRA Works:**
```
Original Weight Matrix (1000x1000):
W = [huge matrix with 1M parameters]

LoRA Decomposition:
W' = W + (A × B)
where:
  A = 1000x8 matrix (8K parameters)
  B = 8x1000 matrix (8K parameters)
  Total: 16K parameters (vs 1M original!)
```

**Benefits:**
- ✅ 99.5% fewer parameters to train
- ✅ Faster training
- ✅ Less memory
- ✅ Can run on consumer hardware
- ✅ Multiple adapters for different tasks

### **38. Model Quantization**
**Definition:** Reducing model precision (e.g., 32-bit floats → 8-bit integers) to save memory and speed up inference with minimal quality loss.

**Example:**
```python
# Full precision (FP32):
weight = 0.123456789  # 32 bits = 4 bytes
Model size: 3.8GB

# 8-bit quantization (INT8):
weight = 31  # 8 bits = 1 byte (maps to ~0.12)
Model size: 1.0GB  # 4x smaller!
```

**Trade-offs:**
```
FP32 (Full Precision):
  ✅ Highest quality
  ❌ 4x more memory
  ❌ Slower inference

INT8 (Quantized):
  ✅ 4x less memory
  ✅ 2-3x faster inference
  ⚠️ ~1-2% quality loss (usually acceptable)
```

**Your Project:** Currently uses FP16 (half precision) on GPU, FP32 on CPU.

### **39. Gradient Descent**
**Definition:** Optimization algorithm used to train neural networks. Iteratively adjusts weights to minimize loss (error).

**Simple Explanation:**
```
Imagine standing on a hill (error surface):
1. Check which direction is downhill (gradient)
2. Take a small step in that direction (learning rate)
3. Repeat until you reach the bottom (minimum loss)
```

**In Training:**
```python
# Simplified training loop
for batch in training_data:
    # Forward pass
    predictions = model(inputs)
    loss = calculate_loss(predictions, targets)
    
    # Backward pass (compute gradients)
    gradients = compute_gradients(loss)
    
    # Update weights (gradient descent)
    for weight in model.weights:
        weight -= learning_rate * gradient  # ← The "descent"
```

**Learning Rate:**
```
Too high (0.1): Steps too big, overshoots minimum
Just right (0.001): Smooth convergence
Too low (0.00001): Training takes forever
```

### **40. Transformer Architecture**
**Definition:** Neural network architecture that powers modern LLMs. Uses self-attention to understand relationships between words.

**Key Innovation - Self-Attention:**
```
Sentence: "The spa at Castle Frankenstein offers massages"

Traditional RNN:
  Processes word-by-word, loses long-range context

Transformer:
  "massages" can attend to "spa", "Castle", "Frankenstein" all at once
  Understands "massages" relates to "spa" more than "at"
```

**Architecture:**
```
Input: "What spa services?"
  ↓
[Token Embedding] (words → numbers)
  ↓
[Positional Encoding] (add position info)
  ↓
[Self-Attention Layers] (12-96 layers)
  │ - Multi-head attention
  │ - Feed-forward networks
  │ - Layer normalization
  ↓
[Output Layer]
  ↓
Output: "Electric Massage, Voltage Relaxation..."
```

**Models Using Transformers:**
- GPT (Generative Pre-trained Transformer) ← You use this
- BERT (Bidirectional Encoder)
- T5 (Text-to-Text Transfer)
- Claude (Anthropic's model)
